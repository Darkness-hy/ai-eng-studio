# 失效模式 — MAST、群体思维、单一文化、级联错误

> 2026 年的参考分类法是 **MAST**（Cemri et al., NeurIPS 2025, arXiv:2503.13657），它源自对 7 个最先进开源多智能体系统（MAS）共 1642 条执行轨迹的分析，结果显示**失败率高达 41–86.7%**。三大根类别：**规格问题（Specification Problems）**（41.77%）——角色模糊、任务定义不清；**协调失效（Coordination Failures）**（36.94%）——通信中断、状态失同步；**验证缺口（Verification Gaps）**（21.30%）——缺少校验、没有质量检查。**群体思维（Groupthink）**家族（arXiv:2508.05687）进一步补充了：单一文化坍塌（同一基础模型 → 错误相关联）、从众偏差（智能体互相强化彼此的错误）、心智理论缺陷、混合动机博弈、级联可靠性失效。级联失效的典型例子：重试风暴——一次支付失败触发订单重试，订单重试又触发库存重试，最终压垮库存服务（几秒内负载翻 10 倍——需要熔断器）。记忆投毒：某个智能体的幻觉进入共享记忆，下游智能体把它当作事实；准确率逐渐衰减，使根因诊断异常痛苦。**STRATUS**（NeurIPS 2025）报告称，通过部署专门的检测 / 诊断 / 验证智能体，缓解成功率提升了 1.5 倍。本课把失效模式当作一等工程目标来对待。

**Type:** Learn
**Languages:** Python (stdlib)
**Prerequisites:** Phase 16 · 13 (Shared Memory), Phase 16 · 14 (Consensus and BFT), Phase 16 · 15 (Voting and Debate Topology)
**Time:** ~75 minutes

## 问题背景

多智能体系统在真实任务上的失败率高达 41-86.7%（Cemri et al. 2025 在 7 个开源 MAS 上测得）。这不是靠"再加几个智能体"就能调通的。这些失败有结构性根源。MAST 分类法给出了类别框架。本课把每个类别映射到具体的检测、诊断和缓解模式，让这些数字不再显得随意。

2026 年的生产实践是把失效模式当作设计输入。在你能逐一指出每个 MAST 类别并说出你部署了什么缓解措施之前，你的架构都谈不上"足够好"。

## 核心概念

### MAST 类别

**规格问题（占失败的 41.77%）。** 智能体的任务没有定义得足够严密。例如：

- 角色模糊：两个智能体都以为自己是评审者。
- 任务规格不足："总结一下这个"，而用户其实想要一个特定角度。
- 成功标准隐含不明：智能体无法判断自己是否完成了任务。

缓解措施：
- 编写显式的角色契约。每个智能体的提示词写明它做什么，*以及它不做什么*。
- 为每个任务定义验收测试。在智能体开工前，先定义"完成的样子是 X"。
- 派发前的规格检查：由一个独立的智能体在任务下发前审查任务定义。

**协调失效（36.94%）。** 通信或状态层面的崩溃。

例如：
- 两个智能体在没有同步机制的情况下更新共享状态。
- 智能体之间的消息丢失（队列故障、超时）。
- 状态漂移：智能体 A 认为任务已完成；智能体 B 还在执行。

缓解措施：
- 带版本号的共享状态，配合乐观并发控制。
- 关键消息要求显式确认（重试直到收到 ack）。
- 周期性的状态同步检查点；及早发现漂移。

**验证缺口（21.30%）。** 对输出没有独立检查。

例如：
- 一个智能体声称成功；没人去核实。
- 智能体链条中每一环都信任上一环的输出。
- 对涌现出的组合行为缺少测试覆盖。

缓解措施：
- 独立的验证者智能体（第 13 课）。只读权限、独立的信息来源。
- 显式的交接契约："A 的输出必须先通过检查器 C，B 才能开始。"
- 记录结果日志以供事后分析。

### 群体思维家族（arXiv:2508.05687）

当智能体趋于同质化或互相模仿时，会出现五种相关的失效：

**单一文化坍塌（Monoculture collapse）。** 相同的基础模型或训练数据 → 错误相关联。当三个智能体共用一个 LLM 时，它们也共享它的幻觉。

**从众偏差（Conformity bias）。** 智能体向声音最大或最自信的同伴靠拢，哪怕对方是错的。

**心智理论缺陷（Deficient ToM）。** 智能体无法建模彼此的信念；协调随之瓦解（第 18 课）。

**混合动机博弈（Mixed-motive dynamics）。** 激励只部分对齐的智能体会漂向折中的中间地带，结果谁都不满意。

**级联可靠性失效（Cascading reliability failures）。** 一个组件的错误模式触发依赖组件的错误模式。

### 级联示例 — 重试风暴

一个经典的 2026 年事故模式：

```
payment service fails 10% of requests
   ↓
order agent retries payment (exponential backoff but naive)
   ↓
each retry is a new order-inventory check
   ↓
inventory service sees 2x normal load
   ↓
inventory service starts timing out
   ↓
every order retries inventory check
   ↓
inventory service sees 10x normal load
   ↓
cluster goes down
```

解法是经典的：**熔断器（circuit breaker）**。当下游错误率超过阈值时，直接短路并返回缓存或默认结果。再加上对每个请求设定重试预算上限。

熔断器是少数几个可以从分布式系统直接照搬、无需改动的多智能体失效缓解手段之一。

### 记忆投毒（再探）

回顾第 13 课：某个智能体的幻觉变成共享记忆中的"事实"，下游智能体基于这个被污染的事实进行推理。用 MAST 的术语来说，这是共享记忆层的验证缺口。

症状是准确率逐渐衰减。你不会看到崩溃，只会看到难以追溯根因的缓慢漂移。

缓解措施：仅追加日志、来源追踪（provenance）、不可写入的验证者。第 13 课已详细介绍。

### STRATUS — 用专门的智能体做失效检测

STRATUS（NeurIPS 2025）报告称，部署以下三类智能体后，缓解成功率提升 1.5 倍：

- **检测智能体。** 监控症状模式（高分歧率、重试激增、准确率漂移）。
- **诊断智能体。** 拿到症状后，根据 MAST 分类法推断可能的根因。
- **验证智能体。** 缓解措施实施后，检查症状是否消除。

这就是 SRE 式的事故响应，应用到了智能体系统上。这三个角色都可以是带专门提示词的 LLM 智能体。

### 失效模式审计

2026 年的最佳实践是每年（或每次大版本发布时）做一次失效模式审计：

1. **采样轨迹。** 收集约 1000 条真实执行轨迹。
2. **归类。** 把每条轨迹中的失败映射到 MAST + 群体思维类别。
3. **计算分类别失败率。** 哪些类别在你的系统中占主导？
4. **给缓解措施排序。** 哪个修复能消除最多的失败？
5. **挑选 2-3 项缓解措施。** 实施；下个季度重新审计。

纪律本身比具体选择更重要。没有审计，失败就会混入噪声，永远得不到系统性解决。

### 当系统静默失效时

最危险的失效类别是静默的正确性失效。一个大声失败的系统（崩溃、异常、告警）可以被监控。而一个产出"看似合理但实际错误"输出的系统，靠异常日志是发现不了的。这就是为什么验证缺口虽然按数量只占 21.30%，却是单次失败代价最高的类别。

值得投入的方向：
- 基于抽样的人工评审。
- 黄金数据集回归测试。
- 对重要输出做跨智能体交叉核验。

### 快失效与慢失效

有些失败是即时的；有些是缓慢的。即时失败（超时、schema 不匹配、鉴权错误）检测成本低。慢失败（记忆投毒、单一文化漂移、角色模糊）检测和预防的成本都很高。

2026 年的工程做法：为慢失效设置代理指标（proxy），在漂移演变成可见错误之前抓住它。一致率、重试率、输出长度分布、相邻智能体版本输出间的编辑距离，都是有用的代理指标。

## 从零实现

`code/main.py` 实现了：

- `FailureTaxonomy` — 把模拟的事故归类到 MAST + 群体思维类别。
- `CircuitBreaker` — 经典模式；错误率超过阈值时断开。
- `RetryStormSimulator` — 演示级联失效；可开关熔断器。
- `DetectionAgent` — 脚本化的 STRATUS 风格症状匹配器。

运行：

```
python3 code/main.py
```

预期输出：
- 无熔断器的重试风暴：库存错误爆炸式增长（模拟）。
- 启用熔断器：在阈值处封顶；返回降级模式的响应。
- 检测智能体标记出该模式，并指出对应的 MAST 类别。

`outputs/skill-mast-auditor.md` 对一个多智能体系统执行 MAST 风格的失效模式审计。轨迹 → 归类 → 缓解措施排序。

## 交付产物

生产环境中的失效模式纪律：

- **每季度做一次 MAST 审计。** 不是每年。随着系统增长，主导类别会变化。
- **熔断器无处不在。** 对任何依赖服务的每个出站调用都加。默认在 5-10% 错误率时断开。
- **黄金数据集。** 小规模、高质量、人工审核过。每周对其做回归测试。
- **STRATUS 三件套。** 检测 + 诊断 + 验证智能体监控生产环境。先只上检测智能体；等症状变得嘈杂时再加诊断。
- **失败预算。** 按类别为失败率设定显式 SLO。超出预算就触发一次"停止发布"的讨论。

## 练习

1. 运行 `code/main.py`。确认熔断器能压制重试风暴。调整失败阈值，观察其中的取舍。
2. 实现一个**慢失效代理指标**：3 个并行智能体之间的一致率。当它骤降时触发告警。通过逐渐让智能体输出相关联来模拟单一文化漂移。
3. 阅读 Cemri et al.（arXiv:2503.13657）。挑选他们研究的 7 个 MAS 系统之一，梳理其排名前 3 的失败类别。这与 MAST 的预测相比如何？
4. 阅读群体思维论文（arXiv:2508.05687）。判断五种模式中哪一种在生产环境中最难检测。提出一个代理指标。
5. 为你熟悉的某个多智能体系统设计一套 STRATUS 风格的检测-诊断-验证三件套。检测智能体监控哪些症状？诊断智能体推荐哪些缓解措施？验证智能体如何确认这些措施有效？

## 关键术语

| 术语 | 大家怎么说 | 实际含义 |
|------|----------------|------------------------|
| MAST | "2026 年的分类法" | Cemri 2025；3 个根类别 + 14 种失败子类型。 |
| 规格问题 | "角色模糊" | 任务或角色定义不足；智能体不知道该做什么。 |
| 协调失效 | "状态漂移" | 智能体之间的通信或同步崩溃。 |
| 验证缺口 | "没人检查过" | 输出未经独立验证就被接受。 |
| 群体思维家族 | "同质化失效" | 单一文化、从众、心智理论缺陷、混合动机、级联。 |
| 单一文化坍塌 | "同一个模型，同样的幻觉" | 共享基础模型或训练数据导致的相关性错误。 |
| 重试风暴 | "级联式错误放大" | 一次失败触发重试，重试放大下游负载。 |
| 熔断器 | "按错误率快速失败" | 错误率超过阈值时断开；用默认结果短路。 |
| STRATUS | "事故响应三件套" | 检测 + 诊断 + 验证智能体。缓解成功率提升 1.5 倍。 |
| 记忆投毒 | "幻觉会传播" | 共享记忆中的事实被污染；下游智能体基于毒数据推理。 |

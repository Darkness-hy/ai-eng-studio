# 角色专业化——规划者、批评者、执行者、验证者

> 这是 2026 年最常见的多智能体分工方式：一个智能体负责规划，一个负责执行，一个负责批评或验证。MetaGPT（arXiv:2308.00352）将其形式化为编码进角色提示词的标准作业程序（SOP）——产品经理、架构师、项目经理、工程师、QA 工程师——遵循 `Code = SOP(Team)` 的公式。ChatDev（arXiv:2307.07924）通过「聊天链」（chat chain）把设计师、程序员、审查者、测试员串联起来，并引入「交流式去幻觉」（communicative dehallucination，智能体显式请求缺失的细节）。验证者是承重角色：Cemri 等人（MAST，arXiv:2503.13657）表明，每一次多智能体失败都能追溯到验证缺失或失效。PwC 报告称，在 CrewAI 中引入结构化验证循环后，准确率提升了 7 倍（10% → 70%）。

**Type:** Learn + Build
**Languages:** Python (stdlib)
**Prerequisites:** Phase 16 · 04 (Primitive Model), Phase 16 · 05 (Supervisor)
**Time:** ~60 minutes

## 问题背景

千篇一律的多智能体系统只能产出千篇一律的结果。把三个写代码的智能体放进群聊，得到的是同一份平庸代码的三种变体。你可以增加更多智能体、增加更多轮次，质量仍然跨不过门槛。

解决之道不是更多的智能体，而是*不同*的智能体。分配各不相同的角色。给批评者配上规划者没有的工具。给验证者一套客观的测试套件。这样系统内部才有了基于事实修正的分歧，而不只是并行的瞎猜。

## 核心概念

### 四个经典角色

**规划者（Planner）。** 阅读目标，产出步骤清单或规格说明。工具：知识检索、文档。输出：结构化计划。

**执行者（Executor）。** 每次读取一个计划步骤，产出工件。工具：实际干活的工具（代码编译器、shell、API 客户端）。输出：工件本身。

**批评者（Critic）。** 对照规划者的意图审阅执行者的输出。工具：对工件的只读访问、静态分析。输出：接受/拒绝及理由。

**验证者（Verifier）。** 读取工件并运行确定性检查。工具：测试运行器、类型检查器、schema 校验器。输出：通过/失败及证据。

批评者是主观的、带观点的，通常基于 LLM。验证者是客观的、确定性的，通常基于代码。它们不是同一个角色。

### MetaGPT 的 SOP 模式

MetaGPT（arXiv:2308.00352）把软件工程的 SOP 编码为角色提示词：

- **产品经理**撰写 PRD。
- **架构师**产出系统设计。
- **项目经理**拆分任务。
- **工程师**负责实现。
- **QA 工程师**运行测试。

每个角色都有严格的输入/输出 schema。角色提示词说明该角色*是什么*以及*必须产出什么*。这就是 `Code = SOP(Team)` 公式——确定性的 SOP 把一组 LLM 变成一条可预测的流水线。

### ChatDev 的交流式去幻觉

ChatDev 增加了一个关键动作：当执行者需要某个计划中没有给出的具体细节时，它会先显式询问设计师再继续。这避免了 LLM 的经典失败模式——煞有介事地编造细节。

实现方式：在角色提示词中加入「当你需要某条未被告知的具体信息时，先按名字向相关角色提问，再产出结果。」

### 为什么验证者最重要

Cemri 等人（MAST）追踪了 1642 次多智能体执行失败。其中 21.3% 是验证缺口——系统交付了一个无人检查过的答案。其余 79% 也往往可以追溯到「存在一个检查，但它静默失败或从未运行」。验证是承重角色。

PwC 报告（2025 年 CrewAI 部署案例）称，加入结构化验证循环后，准确率从 10% 提升到 70%。一个角色带来 7 倍增益。

### 批评者 vs 验证者

- 批评者是审阅工件质量的 LLM。主观。可能被言之凿凿的文字骗过。
- 验证者是对工件运行的确定性程序。客观。给出通过/失败及证据。

两者都要用。批评者能抓住验证者表达不出来的品味问题。验证者能抓住批评者看不见的 bug，因为那些 bug 只在运行时才会暴露。

### 反模式

系统里每个角色都是 LLM，而每个角色的输出都是「我看没问题」。这是经典的 MAST 失败模式。至少加入一个验证者，其通过/失败由代码裁定，而不是由 LLM 裁定。

### 框架对照

- **CrewAI** —— `Agent(role, goal, backstory)` 是教科书式的角色专业化接口。
- **LangGraph** —— 节点可以使用专门的提示词；边强制约束流水线走向。
- **AutoGen** —— 在 GroupChat 中使用名字为单个单词的角色专属 ConversableAgent。
- **OpenAI Agents SDK** —— 在角色专业化的 Agent 之间使用 handoff 工具。

## 从零实现

`code/main.py` 实现了一条构建简单 Python 函数的 4 角色流水线：

- **规划者**产出规格说明。
- **执行者**生成代码字符串。
- **批评者**（用 LLM 模拟）标记明显问题。
- **验证者**在沙箱（`exec`）中针对一个测试用例运行生成的代码。

演示运行两次：第一次执行者产出正确代码（批评者和验证者都通过），第二次执行者产出偏离规格的代码（批评者因为代码看起来合理而漏掉了 bug，验证者因为测试失败而抓住了它）。

运行：

```
python3 code/main.py
```

## 生产实践

`outputs/skill-role-designer.md` 接收一个任务，产出角色名单（3-5 个角色）、每个角色的输入/输出 schema，以及验证者检查项。在把智能体接入框架之前先用它。

## 交付产物

检查清单：

- **至少一个确定性验证者。** 绝不要全 LLM。
- **每个角色有显式 I/O schema。** 规划者返回规格说明而不是散文；执行者按该 schema 读取。
- **交流式去幻觉。** 信息缺失时执行者必须向规划者提问；绝不自行编造。
- **批评者/验证者的顺序。** 先跑批评者（便宜，抓设计问题），后跑验证者（慢，抓 bug）。
- **循环预算。** 批评者-执行者修订最多 2 轮，超过则升级给人类处理。

## 练习

1. 运行 `code/main.py`，观察验证者如何抓住批评者漏掉的 bug。再添加一个静态分析检查（统计 `return` 出现的次数）作为额外的验证者。它能抓住哪些运行时测试漏掉的问题？
2. 添加第 5 个角色：「需求分析师」，负责把用户愿望翻译成规划者可用的规格说明。哪些交流式去幻觉的提问应该上行流向它？
3. 阅读 MetaGPT 第 3 节（"Agents"）。列出 MetaGPT 5 个角色各自的输入/输出 schema。
4. 阅读 ChatDev 的聊天链示意图（arXiv:2307.07924 Figure 3）。找出交流式去幻觉在哪里打断了一个本会无限进行的循环。
5. PwC 的 7 倍准确率增益来自验证循环。假设三个加验证者也无济于事的任务——即对正确性进行确定性检查不可能或代价高得离谱的场景。

## 关键术语

| 术语 | 人们怎么说 | 实际含义 |
|------|----------------|------------------------|
| 角色专业化 | “不同的智能体，干不同的活” | 为规划者/执行者/批评者/验证者角色分别调校的不同系统提示词。 |
| SOP 模式 | “编码进提示词的标准作业程序” | MetaGPT 的提法：每个角色的严格 I/O schema 把一个团队变成一条流水线。 |
| 交流式去幻觉 | “先问再编” | ChatDev 模式：细节缺失时执行者向规划者提问，而不是凭空捏造。 |
| 批评者 | “LLM 审阅者” | 主观、带观点的审阅者。能抓品味问题。可能被言之凿凿的文字骗过。 |
| 验证者 | “确定性检查” | 基于代码的通过/失败判定。测试运行器、类型检查器、schema 校验器。骗不了。 |
| 验证缺口 | “没人检查过” | 占 MAST 失败的 21.3%。答案在没有经过本可抓住 bug 的检查的情况下被交付。 |
| 修订循环 | “批评者打回重做” | 批评者的拒绝触发执行者带反馈重跑。需要预算限制。 |
| 全 LLM 反模式 | “我看没问题” | 每个角色都是 LLM，没有确定性检查。经典 MAST 失败。 |

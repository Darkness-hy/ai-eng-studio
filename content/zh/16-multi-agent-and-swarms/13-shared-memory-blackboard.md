# 共享内存与黑板模式

> 2026 年的多智能体系统中并存着两种方案：**消息池（message pool）**（所有人都能看到所有消息，如 AutoGen GroupChat 或 MetaGPT）和**带订阅机制的黑板（blackboard）**（智能体订阅相关事件，如 Context-Aware MCP 或 Matrix 框架）。这两者都是多智能体系统中唯一有状态的部分——这意味着有趣的 bug 都藏在这里。最典型的失败模式是**记忆投毒（memory poisoning）**：一个智能体幻觉出一条「事实」，其他智能体把它当作已验证的信息，准确率逐渐衰减——这种问题比立即崩溃难调试得多。本课用标准库从零构建这两种结构，注入一次投毒攻击，并演示生产环境中真正有效的三种缓解手段。

**Type:** Learn + Build
**Languages:** Python (stdlib, `threading`)
**Prerequisites:** Phase 16 · 04 (Primitive Model), Phase 16 · 09 (Parallel Swarm Networks)
**Time:** ~75 minutes

## 问题背景

多智能体系统需要一个让智能体共享事实的地方。最直接的做法是「把所有内容都塞进消息里传递」——但这等于用额外的拷贝重新发明了共享状态。另一种是「给所有人一份全局日志」——但全局日志会无限增长，而且极易被投毒。第三种是「为每个智能体投影一个视图」——可扩展性好，但需要大量 schema 设计。

当其中一个智能体产生幻觉并把幻觉写入共享状态时，每个读取该状态的下游智能体都会把幻觉当作事实采纳。等到人类发现问题时，推理链已经深入五步，而根因是第三条写入的消息。调试多智能体的准确率衰减比调试崩溃难得多。

这就是记忆投毒。它是 MAST 分类法（Cemri et al., arXiv:2503.13657）中记录第二多的失败族群，而且它是结构性的：任何没有溯源机制和不可写验证者的共享内存设计，最终都会出现这个问题。

## 核心概念

### 两种主要拓扑

**全量消息池。** 每个智能体读取每条消息。AutoGen GroupChat 和 MetaGPT 采用这种方式。简单、透明、易于审查，但无法扩展到约 10 个智能体以上，因为每个智能体的上下文会被其他智能体的工作填满。

```
agent-A ──write──▶ ┌────────────────┐ ◀──read── agent-D
                   │ message pool   │
agent-B ──write──▶ │                │ ◀──read── agent-E
                   │ (global log)   │
agent-C ──write──▶ └────────────────┘ ◀──read── agent-F
```

**带订阅机制的黑板。** 智能体声明自己对哪些主题感兴趣；底层基座只路由相关消息。CA-MCP（arXiv:2601.11595）和去中心化框架 Matrix（arXiv:2511.21686）采用这种方式。扩展性更好，但需要预先设计 schema，订阅才有意义。

```
                   ┌─ topic: prices ──┐
agent-A ──pub────▶ │                  │ ──▶ agent-D (subscribed)
                   ├─ topic: orders ──┤
agent-B ──pub────▶ │                  │ ──▶ agent-E (subscribed)
                   ├─ topic: alerts ──┤
agent-C ──pub────▶ │                  │ ──▶ agent-F (subscribed)
                   └──────────────────┘
```

### 各自的适用场景

- **全量消息池**适合智能体数量少（< 10）、角色异构、对话周期短的场景。当所有人都能看到一切时，推断「谁说了什么」轻而易举。
- **黑板**适合智能体数量多、角色同质但实例众多（集群/swarm）、对话长期运行的场景。路由机制能节省 token 成本并避免上下文污染。

生产系统常常混用两者：顶层（规划层）用一个小型全量消息池，下层（执行层）用黑板。

### 一个记忆投毒场景

三个智能体协作完成一项研究任务。智能体 A 是检索智能体，智能体 B 是摘要器，智能体 C 是分析师。

1. A 抓取一个网页并向共享状态写入一条消息：「该研究报告了 42% 的准确率提升。」
2. 网页实际写的是「4.2% 的提升」。A 幻觉掉了一个小数点。
3. B 读取共享状态后写入：「报告了高达 42% 的准确率增益（来源：A）。」
4. C 读取共享状态后写入：「建议采纳——42% 的提升是革命性的。」
5. 最终报告引用了一个从未存在过的 42%。

没有智能体崩溃。没有测试失败。系统「正常运行」。幻觉借助共享状态从一个智能体的上下文渗透进了每个下游智能体的推理。

### 为什么这是结构性问题

没有共享状态时，智能体 A 的幻觉只停留在 A 的上下文里。下游智能体会重新抓取或重新推导，有可能发现错误。而有了朴素的共享状态后，A 的上下文变成了所有人的上下文，幻觉被「洗白」成了事实。

问题不在于共享状态本身——而在于**缺乏溯源机制、缺乏独立验证者**的共享状态。三种缓解手段可以解决这个问题：

1. **为每次写入附加溯源信息。** 共享状态中的每个条目都记录是谁写的、何时写的、在什么提示词下写的，以及（如适用）智能体引用了什么来源。下游智能体根据溯源信息调整自己读取时的怀疑程度。
2. **写入带版本号；只允许追加（append-only）。** 修正是一个取代旧条目的新条目，而不是原地更新。审计轨迹得以保留。
3. **至少保留一个无法写入共享状态的智能体。** 一个只读的验证者智能体抽样检查条目、重新抓取来源并标记不一致。因为它无法写入消息池，所以不会被消息池投毒。

### 黑板模式的先例（Hayes-Roth, 1985）

黑板模式比 LLM 智能体早出现四十年。Hayes-Roth（1985, "A Blackboard Architecture for Control"）描述了一组专家知识源（Knowledge Source），它们观察一块全局黑板、贡献部分解、并触发其他知识源。2026 年的黑板（CA-MCP、Matrix）就是同一种模式，只是知识源换成了 LLM 智能体，部分解换成了 JSON 数据块。写入争用、机会式控制、一致性这些问题，旧文献里都有成文的解决方案，现代系统正在重新发现它们。

### 投影视图与全量视图

纯黑板给每个订阅者同样的投影（按主题划分）。更激进的设计是**按智能体投影（per-agent projection）**：每个智能体得到一个为其角色定制的视图。LangGraph 的状态 reducer 是 2026 年的典型实现——reducer 函数把全局状态折叠成角色专属的切片。

按智能体投影扩展性更好，但需要 schema。没有 schema，你就得在每个智能体的提示词里临时重建投影逻辑。

### 写入争用模式

多个智能体同时写入是一个并发问题，不只是 LLM 问题。三种模式行之有效：

- **顺序写入者（单生产者）。** 所有写入都经过一个协调智能体串行化。简单，但会成为瓶颈。
- **带版本号的乐观并发。** 每个条目都有版本号；写入者在版本不匹配时失败并重试。经典的数据库技术。
- **主题分区。** 不同智能体拥有不同主题。跨主题不存在争用。需要预先设计分区边界。

2026 年的多数框架默认采用顺序写入者，因为 LLM 调用足够慢，争用很少发生，瓶颈也不构成实际损害。

### 不可写验证者

最关键的缓解手段是只读验证者。实现规则：

- 验证者与团队共享状态（可以读取黑板或消息池）。
- 验证者没有共享状态的写入句柄——只能写入一个独立的验证通道。
- 验证者独立抓取写入中引用的来源，标记不一致之处。
- 验证者自己的输出被路由给人类或一个独立的决策智能体，绝不回流到消息池。

没有这种隔离，验证者的输出会变成消息池中的新条目，这意味着被投毒的消息池会投毒验证者，进而投毒它的验证结果。

## 从零实现

`code/main.py` 用标准库 Python 实现了两种拓扑，外加一个玩具级投毒攻击和三种缓解手段。

- `MessagePool` —— 线程安全的追加日志，支持全量读出。
- `Blackboard` —— 按主题键控的发布/订阅，支持按智能体订阅。
- `ProvenanceEntry` —— 每次写入记录 (writer, timestamp, prompt_hash, source_uri)。
- `PoisoningScenario` —— 运行一个三智能体研究任务，其中智能体 A 幻觉掉一个小数点。打印最终报告。
- `Verifier` —— 一个只读智能体，重新抓取来源并标记不一致。在验证者在场的情况下运行同一场景。

运行：

```
python3 code/main.py
```

预期输出：
- 第 1 次运行（无验证者）：幻觉出的 42% 一路传播进最终报告。
- 第 2 次运行（有验证者）：验证者标记出不一致，消息池被打上「flagged」标签，最终报告包含一条撤回声明。

## 生产实践

`outputs/skill-memory-auditor.md` 是一个技能，可审计任意多智能体系统的共享内存设计在溯源、版本化和验证者隔离方面的表现。在新的多智能体架构上线生产前先用它跑一遍。

## 交付产物

对任何共享内存设计：

- 每次写入都记录溯源信息：`(writer, timestamp, prompt_hash, tool_calls_cited, source_uri)`。
- 让日志只允许追加。修正是引用被取代条目的新条目。
- 部署至少一个具备独立来源访问能力的只读验证者智能体。
- 把验证者的输出路由到独立通道，不要回流到共享消息池。
- 记录「取代型写入」占总写入的比例——比例上升是幻觉模式的早期证据。

## 练习

1. 运行 `code/main.py`。确认第 1 次运行传播了幻觉，第 2 次运行捕获了它。
2. 添加第二个幻觉：智能体 B 编造一个数据集规模。验证者应该在没有针对任何一个幻觉手工调参的情况下同时捕获两者。
3. 把全量消息池换成带主题分区（`prices`、`summaries`、`analyses`）的黑板。主题分区让哪些投毒场景更难得手？又对哪些场景无济于事？
4. 阅读 Hayes-Roth（1985, "A Blackboard Architecture for Control"）。找出论文中本课没有讨论、但 2026 年的系统能从中受益的两种控制模式。
5. 阅读 CA-MCP（arXiv:2601.11595）。把它的 Shared Context Store 映射到 `code/main.py` 中的 MessagePool 或 Blackboard 类。CA-MCP 在此之上额外添加了哪些原语？

## 关键术语

| 术语 | 常见说法 | 实际含义 |
|------|----------------|------------------------|
| 消息池（Message pool） | 「共享聊天记录」 | 每个智能体都读取的追加日志。完全透明，扩展性差。 |
| 黑板（Blackboard） | 「共享工作区」 | 按主题键控的发布/订阅。智能体订阅相关主题。扩展性更好。 |
| 溯源（Provenance） | 「谁写了什么」 | 每次写入的元数据：写入者、时间戳、提示词、来源。 |
| 记忆投毒（Memory poisoning） | 「幻觉扩散」 | 一个智能体的错误进入共享状态，下游智能体把它当作事实采纳。 |
| 只允许追加（Append-only） | 「不做原地更新」 | 修正是取代旧条目的新条目。保留审计轨迹。 |
| 不可写验证者（Unwritable verifier） | 「独立审计员」 | 重新抓取来源并标记不一致的只读智能体。 |
| 投影（Projection） | 「限定范围的视图」 | 从全局状态计算出的按智能体视图。LangGraph reducer 是典型案例。 |
| 知识源（Knowledge Source） | 「专家智能体」 | Hayes-Roth 1985 年对黑板参与者的称呼。 |

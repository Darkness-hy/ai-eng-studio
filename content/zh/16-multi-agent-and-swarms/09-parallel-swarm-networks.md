# 并行 / 蜂群 / 网络化架构

> 与监督者模式形成对照：没有中心决策者。智能体读取共享事件总线，异步领取任务，再把结果写回。LangGraph 明确支持面向去中心化、动态环境的「蜂群架构」（Swarm Architecture）。Matrix（arXiv:2511.21686）把控制流和数据流都表示为经分布式队列传递的序列化消息，从而消除编排器瓶颈。其中的取舍是明确的：用确定性和可追溯性换取可扩展性。蜂群适合包含大量独立子问题的任务；不适合需要单一连贯计划的任务。

**Type:** Learn + Build
**Languages:** Python (stdlib, `threading`, `queue`)
**Prerequisites:** Phase 16 · 05 (Supervisor Pattern), Phase 16 · 04 (Primitive Model)
**Time:** ~75 minutes

## 问题背景

监督者模式能撑到几个 worker 的规模。但要是几百个呢？监督者本身会变成瓶颈：每一个「谁去做什么」的决策都要经过这一个智能体。一个缓慢的规划步骤就能拖住整个系统。

蜂群（swarm）架构把设计颠倒过来。不再由中心规划者分派工作，而是让 worker 从共享队列里自己取任务。「协调」被内化到事件总线的语义里。没有编排器；只要队列扛得住，系统就能继续扩展。

## 核心概念

### 整体形态

```
                ┌──── shared queue ────┐
                │                      │
       ┌────────┼────────┐  ◄──────┬───┘
       ▼        ▼        ▼         │
     Worker  Worker  Worker   Worker
      A       B       C        D
       │        │        │         │
       └────────┴────────┴─────────┘
                 │
                 ▼
            results pool
```

没有编排器。每个 worker 循环往复：取一个任务，处理，写回结果（也可以顺手把后续任务入队）。

### 蜂群适用的场景

- **大量相互独立的任务。** 抓取、转换、分类。任务之间没有依赖。
- **耗时不均的工作。** 如果有的任务跑 100ms、有的跑 10s，蜂群会自动均衡负载——快的 worker 直接去拉下一个活。监督者却必须提前预估耗时。
- **吞吐量优先于确定性。** 你关心的是总完成时间，而不是严格的执行顺序。

### 蜂群失效的场景

- **有顺序要求的工作流。** 如果第 3 步需要第 2 步的输出，蜂群可能在第 2 步完成之前就触发第 3 步。
- **需要全局规划的任务。** 复杂的研究问题需要一个规划者。一群各自为战的研究型智能体产出的是零散事实，而不是一份连贯的报告。
- **调试。** 没有中心日志、工作又是异步的，复现一个 bug 的代价很高。

### Matrix（arXiv:2511.21686）

Matrix 是 2025 年的一篇论文，它把蜂群思路推到了逻辑终点：控制流和数据流全部是分布式队列上的序列化消息。没有中心协调者。容错来自消息的持久性。可扩展性是消息代理（message broker）的问题，而不是系统本身的问题。

它的贡献在于：提出了一种编程模型，多智能体协调变成了「这个智能体订阅哪个消息主题？」，而不是「监督者下一个挑哪个智能体？」。这让整个系统看起来像一张发布/订阅（pub/sub）事件网格。

### LangGraph 的蜂群架构

LangGraph 2025 年的文档明确把「蜂群架构」列为多智能体模式之一：智能体是节点，但边构成一张带环的有向图，池中的任意节点都可以被激活。worker 按条件从可用工作中挑选，而不是听监督者指派。

### 失效模式：饥饿与热点

如果所有 worker 都优先拉取最快可完成的任务，长耗时任务就一直没人领，直到它们成为队列里仅剩的任务。这是经典的队列饥饿（starvation）问题。

缓解手段：

- 带显式老化（aging）机制的优先级队列（等待时间越长优先级越高）。
- worker 特化：部分 worker 只接「长」任务。
- 背压（back-pressure）：限制进入队列的快任务数量。

### 与基于内容路由的衔接

蜂群与基于内容的路由（第 22 课）天然契合。不用一个通用队列，而是每种消息类型各设一个队列。专才 worker 只订阅自己负责的类型。这正是能扩展到上千个智能体的消息总线架构的基础。

## 从零实现

`code/main.py` 实现了一个由 4 个 worker 线程组成的蜂群，它们从一个共享的 `queue.Queue` 拉取任务。任务耗时不均（有快有慢）。演示对比了三种方式：

- **顺序基线：** 单个 worker 串行处理所有任务。
- **固定分配：** 每个任务预先指派给特定 worker（监督者风格）。
- **蜂群：** worker 从共享队列自行拉取。

蜂群自动均衡负载；固定分配则会让快的 worker 在分到慢任务的 worker 干活时干等。

运行：

```
python3 code/main.py
```

输出展示每个 worker 处理的任务数（蜂群的分布不均匀但最优）以及实际耗时（wall-clock time）。

## 生产实践

`outputs/skill-swarm-fit.md` 用于评估一个任务该用蜂群还是监督者。输入项：任务独立性、耗时方差、顺序要求、可调试性需求。

## 交付产物

检查清单：

- **带老化机制的优先级队列。** 防止长任务饥饿。
- **worker 幂等性。** 如果某个 worker 中途崩溃，同一任务可能被拉取多次。worker 必须幂等。
- **持久化队列。** 生产环境用 Kafka、Redis Streams 或数据库支撑的队列。`queue.Queue` 只存在于内存中。
- **任务级可观测性。** 每个任务都有 trace ID；每个 worker 在开始/结束时带着它打日志。
- **背压。** 如果队列的增长速度超过 worker 的消费速度，就让生产者放慢。

## 练习

1. 运行 `code/main.py`。在耗时不均的负载下，蜂群比顺序执行快多少？比固定分配快多少？
2. 增加一个优先级队列变体（使用 `queue.PriorityQueue`）。按任务的 "importance" 字段赋优先级。观察在持续负载下低优先级任务是否会饿死。
3. 实现一个热点检测器：当任意 worker 处理的任务数达到最慢 worker 的 3 倍时打日志。这说明任务耗时分布有什么特点？
4. 阅读 Matrix 论文（arXiv:2511.21686）的摘要和第 3 节。找出 Matrix 接受的一项具体取舍（换来的可扩展性）和它放弃的一项（可追溯性、确定性）。
5. 把蜂群演示改成使用存放 (task_type, payload) 元组的 `queue.Queue`，让 worker 只订阅特定类型。当任务类型异构时，什么样的路由规则才合理？

## 关键术语

| 术语 | 人们怎么说 | 实际含义 |
|------|----------------|------------------------|
| 蜂群架构 | 「去中心化的智能体」 | worker 从共享队列拉取任务；没有中心编排器。 |
| 事件总线 | 「智能体订阅主题」 | 按类型或内容把任务路由给 worker 的消息代理。 |
| 饥饿 | 「任务永远轮不到」 | 低优先级任务因为高优先级工作源源不断到来而一直没人领。 |
| 热点 | 「一个 worker 被淹没」 | 负载失衡，大多数任务都落到同一个 worker 上。 |
| 背压 | 「让生产者慢下来」 | 队列被填满时向上游发出停止生产信号的机制。 |
| 幂等 worker | 「重跑也安全」 | 同一任务处理两次产生相同结果。worker 可能中途崩溃，因此这是必需的。 |
| 持久化队列 | 「崩溃也不丢」 | 由磁盘或多副本存储支撑的队列；worker 崩溃时任务不会丢失。 |
| Matrix 框架 | 「全消息传递的蜂群」 | 数据流和控制流都是分布式队列上的序列化消息。 |

# 层级式架构及其失效模式

> 层级式（hierarchical）就是嵌套的监督者：管理者智能体之下是子管理者，子管理者之下是工作者。CrewAI 的 `Process.hierarchical` 是教科书式的实现：由一个 `manager_llm` 动态分派任务并校验输出。LangGraph 的对应写法是 `create_supervisor(create_supervisor(...))`。当任务本身就是一张真实的组织架构图时，这是最自然的模式。但它也是最容易陷入"管理层空转"的模式——管理者智能体分派任务不当、误读下级输出、或迟迟无法达成共识。很多时候顺序式（sequential）反而更胜一筹。

**Type:** Learn + Build
**Languages:** Python (stdlib)
**Prerequisites:** Phase 16 · 05 (Supervisor Pattern)
**Time:** ~60 minutes

## 问题背景

一旦理解了监督者模式，自然会想到下一步："如果工作者本身也是监督者呢？"团队有子团队，公司有部门套部门。层级式架构正是对这种结构的映射。

问题在于：LLM 管理者和人类管理者并不一样。人类管理者对下属的能力有稳定的先验认知，而 LLM 管理者每一轮都要根据上下文中的内容重新推理整个组织结构。上下文只要出现细微漂移，整棵树的任务分配就会出错。

## 核心概念

### 结构形态

```
                 Manager
                 ┌─────┐
                 └──┬──┘
           ┌────────┴────────┐
           ▼                 ▼
       Sub-Mgr A         Sub-Mgr B
       ┌─────┐           ┌─────┐
       └──┬──┘           └──┬──┘
         ┌┴──┬──┐          ┌┴──┐
         ▼   ▼  ▼          ▼   ▼
       W1  W2  W3         W4  W5
```

每个内部节点都负责规划、分派和汇总，只有叶子节点才真正干活。

### 适用场景

- **组织结构映射清晰。** 如果真实任务本身就是按部门划分的（"法务审一遍文档，财务审一遍文档，工程审一遍文档，最后给高管做总结"），层级结构是显而易见的。
- **局部汇总。** 每个子管理者先把自己团队的输出汇总好，再交给顶层管理者。顶层管理者只看到三份子管理者摘要，而不是十五份工作者输出。

### 失效场景

2026 年的各类事故复盘反复发现三种失效模式：

1. **任务分派错误。** 管理者读完目标后，凭空臆造出一种任务分解方式，把任务派给了错误的子管理者。由于子管理者会老老实实地执行分到手的任务，错误只会在顶层汇总时才暴露出来——而那已经离人类本可以发现问题的位置隔了一层。
2. **输出误读。** 子管理者返回"无法核实论断 X"，顶层管理者却总结成"论断 X 未获确认"。语义在每一层都在漂移。
3. **共识循环。** 两个子管理者意见不一；顶层管理者要求它们协调；它们再向下重新分派；工作者重新执行；子管理者返回略有差异的新答案；循环往复。CrewAI 的 `Process.hierarchical` 用步数上限来防范这种情况，但这个上限本身又成了一个新的超参数。

### 决定性问题

顺序式（线性流水线）还是层级式：你的任务是真的存在相互独立的子团队，还是一条线性流程伪装成了一棵树？如果是后者，用顺序式。如果是前者，可以用层级式，但要预先制定明确的协调规则。

### CrewAI 的实现

`Process.hierarchical` 在多个专家 crew 之上接入一个管理者 LLM。该管理者：

- 接收顶层任务，
- 向各 crew 分派子任务，
- 评估各 crew 的输出，
- 决定是接受、重新分派，还是再迭代一轮。

文档：https://docs.crewai.com/en/introduction（在 Core Concepts 下查找 "Hierarchical Process"）。

### LangGraph 的实现

LangGraph 使用嵌套的 `create_supervisor` 调用。内层监督者拥有自己的图；外层监督者把内层的图当作一个不透明节点来对待。这种方式在调试上比 CrewAI 更清爽（你可以分别单步调试每张图），但更难表达对树结构的动态重组。

参考：https://reference.langchain.com/python/langgraph-supervisor。

## 从零实现

`code/main.py` 运行一个三层的层级结构：

- 顶层管理者：把任务拆成"工程"和"法务"两个分支，
- 工程子管理者：再拆成"前端"和"后端"两个工作者，
- 法务子管理者：只有一个工作者。

演示对比了正常路径（各方一致）和一条**扰动路径**：顶层管理者在任务分解时把"法务"误标成了"财务"，然后观察错误如何层层级联——子管理者老老实实地做了财务方面的工作，顶层汇总者报告的是财务结论，而最初的法务问题无人回答。

运行：

```
python3 code/main.py
```

输出会展示两条路径，并清晰地并排对比"用户问的是什么"与"实际交付了什么"。

## 生产实践

`outputs/skill-hierarchy-fitness.md` 用于评估给定任务应该采用层级式、顺序式还是扁平监督者模式。输入：任务描述、组织结构、协调预算。输出：模式推荐，以及需要重点防范的具体失效模式。

## 交付产物

如果你要将层级式架构投入生产：

- **树深度上限设为 2。** 三层结构已经会让大部分错误逃出可观测性的覆盖范围。
- **明确的协调预算。** 设定顶层管理者必须拍板前的最大协调轮数，通常为 2。
- **每次汇总都带溯源信息。** 每个节点的摘要必须注明它由哪些叶子输出生成。
- **对分解漂移设置告警。** 逐步记录管理者的任务分解结果，与用户查询做差异比对。一旦分解不再覆盖原始查询，立即触发告警。

## 练习

1. 运行 `code/main.py`，对比正常路径与扰动路径。经过多少层管理者交接之后，顶层输出会与用户的问题完全脱节？
2. 增加第三层（顶层 → 子层 → 子子层 → 工作者）。测量随着深度增加，扰动路径自我纠正与完全偏离的比例各是多少。
3. 在每个子管理者下实现一个"金丝雀"工作者，始终原封不动地向它提出用户最初的问题。用金丝雀的答案来检测分解漂移。当金丝雀的答案与汇总答案不一致时，管理者应该如何应对？
4. 阅读 CrewAI 的 `Process.hierarchical` 文档。找出 CrewAI 应用的一个具体护栏（步数上限、manager_llm 约束），并说明它针对的是哪种失效模式。
5. 对比嵌套的 LangGraph 监督者与 CrewAI 层级模式。哪一种能让协调循环更容易被检测出来？

## 关键术语

| 术语 | 大家怎么说 | 实际含义 |
|------|----------------|------------------------|
| 层级式（Hierarchical） | "组织架构图模式" | 监督者之上还有监督者；只有叶子节点干活。 |
| 管理者 LLM（Manager LLM） | "老板" | 在内部节点上负责分解、分派和校验的 LLM。 |
| 分解漂移（Decomposition drift） | "老板跑题了" | 顶层管理者的任务拆分不再覆盖原始问题。 |
| 协调循环（Reconciliation loop） | "没完没了的会议" | 子管理者意见不一；顶层重新分派；工作者重新执行；循环直到预算耗尽。 |
| 深度 2 上限（Depth-2 ceiling） | "别超过两层" | 经验性护栏：三层及以上会让可观测性崩塌。 |
| 金丝雀问题（Canary question） | "每一层都有基准真值" | 始终被原封不动地问到原始查询的工作者，用于检测漂移。 |
| 溯源链（Provenance chain） | "谁说了什么" | 从每次汇总回溯到生成它的叶子输出的追踪链路。 |

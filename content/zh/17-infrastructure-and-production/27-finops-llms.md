# LLM 的 FinOps —— 单位经济学与多租户成本归因

> 传统 FinOps 在 LLM 开销上会失灵。成本是按 token 计费的交易，而不是按资源运行时长计费。标签体系套不上去——一次 API 调用是一笔交易，不是一项资产。工程决策（提示词设计、上下文窗口、输出长度）就是财务决策。2026 年的实战手册要求从第一天起就埋点三个归因维度：按用户（`user_id`）服务于席位定价和扩容洽谈，按任务（`task_id` + `route`）服务于产品功能面的成本核算与优先级排序，按租户（`tenant_id`）服务于单位经济学和续约。四个 token 层——提示、工具、记忆、响应——混进一个桶会掩盖开销。多租户产品的强制执行阶梯：按租户限流（预期峰值的 2-3 倍，明确返回 429 + retry-after）；每日开销上限（合同上限的 1.5-3 倍；触发后收紧限流 + 告警）；开销 z 分数 > 4 时触发熔断开关（kill switch）（自动暂停 + 呼叫值班）。归因模式：打标聚合、遥测关联（trace-ID → 账单；精度最高）、采样外推、基于模型的分摊、事件溯源、实时流式。单位指标：每解决一次查询的成本、每生成一个产物的成本——而不是 $/M tokens。事后补打标签总会有遗漏；要在请求创建时就埋点。

**Type:** Learn
**Languages:** Python (stdlib, toy cost-attribution simulator with kill switch)
**Prerequisites:** Phase 17 · 13 (Observability), Phase 17 · 14 (Caching)
**Time:** ~60 minutes

## 学习目标

- 解释为什么传统 FinOps（标签 + 套餐层级）在 LLM 开销上会失灵，并说出三个新的归因维度。
- 列举四个 token 层（提示、工具、记忆、响应），以及为什么单桶计费会掩盖成本。
- 为多租户产品设计强制执行阶梯（限流 → 开销上限 → 熔断开关）。
- 选择单位指标（每解决一次查询 / 每生成一个产物的成本），而不是 $/M tokens。

## 问题背景

账单上写着 $40,000。但你不知道：
- 是哪个租户花掉的。
- 是哪个产品功能驱动的。
- 是否有个别用户在滥用。
- 罪魁祸首是提示词膨胀、工具调用，还是记忆放大。

供应商侧的打标聚合对云资源（EC2、S3）有效，因为标签会传播到账单明细。LLM API 调用不会自动打标——你必须在调用点打上 user/task/tenant 标记并全程透传。事后归因总会漏掉边角情况。

## 核心概念

### 三个归因维度

**按用户**（`user_id`）：谁花了多少钱。驱动席位定价、扩容洽谈，识别重度用户。

**按任务**（`task_id` + `route`）：哪个产品功能面花了多少钱。驱动功能优先级排序，以及砍掉高成本功能的决策。

**按租户**（`tenant_id`）：哪个客户是盈利的。驱动单位经济学、续约定价、套餐层级阈值。

从第一天起就在调用点对三个维度全部埋点。事后补救永远更糟。

### 四个 token 层

| 层 | 示例 | 占总量的典型比例 |
|-------|---------|---------------------|
| 提示 | system + 用户输入 | 40-60% |
| 工具 | 回灌的工具调用结果 | 20-40%（智能体负载） |
| 记忆 | 历史对话 / 检索到的文档 | 10-30% |
| 响应 | 模型输出 | 10-30% |

把四层混在一个桶里，优化就无从下手。在归因 schema 中把它们拆开。

### 强制执行阶梯

1. **限流**，按租户。预期峰值的 2-3 倍。返回 429 并附带 `Retry-After`。租户会感到受限，但不会收到意外账单。

2. **每日开销上限**，按租户。合同上限的 1.5-3 倍。触发动作：收紧限流 + 告警客户成功团队。

3. **熔断开关**：相对租户基线的开销 z 分数 > 4 时触发。自动暂停该租户；呼叫值班；上报运维 + 客户成功。

### 归因模式

- **打标聚合（tag-and-aggregate）**：在元数据头打标；事后聚合。简单；粗糙。
- **遥测关联（telemetry joiner）**：通过 trace ID 把链路追踪与账单关联。精度最高。成熟团队的做法。
- **采样 + 外推**：采样 5-10%，再按比例放大。粗略估算开销时性价比高；会漏掉长尾。
- **基于模型的分摊**：用回归推断成本驱动因素。适用于没有标签的历史数据。
- **事件溯源（event-sourced）**：把成本作为事件写入流（Kafka / Kinesis）。实时。
- **实时流式**：仪表盘亚秒级更新。

### Cost per X 才是单位指标

$/M tokens 是供应商话术。产品指标应该是：

- 每解决一张支持工单的成本。
- 每生成一篇文章的成本。
- 每完成一次成功的智能体任务的成本。
- 每用户会话分钟的成本。

把成本和产品结果绑定。否则优化就没有锚点。

### 成本归因 trace 的形态

```
trace_id: abc123
  user_id: u_42
  tenant_id: t_7
  task_id: task_classify_doc
  route: model_haiku
  layers:
    prompt_tokens: 1800
    tool_tokens: 600
    memory_tokens: 400
    response_tokens: 150
  cost_usd: 0.0135
  cached_input: true
  batch: false
```

每次调用都发出。存入数据湖。按维度聚合。这部分就落在 Phase 17 · 13 的可观测性技术栈里。

### 复合节省叠加栈

叠加项：缓存 + 批处理 + 路由 + 网关。四项齐备时：
- 缓存 L2（Phase 17 · 14）：输入便宜约 10 倍。
- 批处理（Phase 17 · 15）：五折。
- 路由到便宜模型（Phase 17 · 16）：成本降低 60%。
- 网关效率（Phase 17 · 19）：冗余 + 重试。

最佳叠加情况：约为朴素基线的 5-10%。大多数团队只启用了 2-3 个杠杆；很少有人把四个全部叠上。

### 你应该记住的数字

- 归因维度：按用户、按任务、按租户。
- 四个 token 层：提示、工具、记忆、响应。
- 熔断开关：开销 z 分数 > 4。
- 单位指标：每解决一次查询的成本，而不是 $/M tokens。
- 叠加优化：有可能做到约为基线的 5-10%。

## 关键术语

| 术语 | 大家怎么说 | 实际含义 |
|------|----------------|------------------------|
| 按用户归因 | “用户级成本” | 每次调用都打上 `user_id` |
| 按任务归因 | “功能成本” | `task_id` + `route` 标识产品功能面 |
| 按租户归因 | “客户成本” | `tenant_id`；驱动单位经济学 |
| 四个 token 层 | “成本分层” | 提示 + 工具 + 记忆 + 响应 |
| 限流 | “429 守门” | 在网关强制执行的按租户上限 |
| 每日开销上限 | “每日封顶” | 租户级预算并附带告警 |
| 熔断开关 | “自动暂停” | 开销 z 分数 > 4 触发自动停用 |
| 每解决一次的成本 | “产品单位指标” | 成本绑定产品结果，而非 token |
| 遥测关联 | “trace 对账单” | 精度最高的归因模式 |
| 叠加优化 | “缓存+批处理+路由+网关” | 复合节省至约基线的 5-10% |

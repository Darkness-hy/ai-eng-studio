# 批量 API（Batch API）——成为行业标准的五折优惠

> 每家主流厂商都提供异步批量 API（Batch API），统一打五折，承诺约 24 小时内返回结果。OpenAI、Anthropic、Google 以及大多数推理平台（Fireworks 的 batch 档位、Together 的 batch）实现的都是同一套模式。把批量与提示词缓存（prompt caching）叠加使用，过夜流水线的成本能降到同步无缓存方案的约 10%。规则简单得近乎粗暴：只要不是交互式任务，就该走批量。内容生成流水线、文档分类、数据抽取、报告生成、批量标注、商品目录打标——任何能容忍 24 小时延迟的工作负载，不迁到批量上就是把钱白白丢在桌上。2026 年的生产实践是把每个新的 LLM 工作负载分流到三条通道：交互式（同步调用加缓存）、半交互式（异步队列加降级回退）、批量（过夜执行，叠加缓存输入）。那些伪装成交互式、实际能容忍数分钟延迟的工作负载，浪费最严重。

**Type:** Learn
**Languages:** Python (stdlib, toy batch-vs-sync cost simulator)
**Prerequisites:** Phase 17 · 14 (Prompt & Semantic Caching)
**Time:** ~45 minutes

## 学习目标

- 说出三大厂商的批量 API（OpenAI、Anthropic、Google），以及共同的五折优惠与 24 小时返回保证。
- 计算在过夜分类工作负载上叠加批量与缓存输入的成本，并与同步无缓存基线对比。
- 把一个工作负载分流到交互式 / 半交互式 / 批量三条通道之一，并说明理由。
- 说出两个陷阱：部分交互性（用户期望快于 24 小时）和输出格式漂移（批量文件格式因厂商而异）。

## 问题背景

你的团队上线了一条夜间报告生成流水线：5 万份文档，逐份摘要，对摘要做聚类，再起草一份高管简报。同步运行需要 4 小时，每晚花费 2,000 美元。这时你听说了批量 API。

批量直接打五折。你又在系统提示词上启用了提示词缓存（全部 5 万次调用共享同一段系统提示词）。两者叠加后，账单降到每晚 180 美元——约为基线的 9%。同一条流水线，只改了三处配置。

批量是 LLM 成本工具箱里最便宜却没人去拉的那根杠杆。原因大多出在组织层面：团队习惯性地认为需要"实时"，而真正的 SLA 其实只是"第二天早上之前"。这节课要解决的，就是别把 90% 的账单白白留在桌上。

## 核心概念

### 三大批量 API

**OpenAI Batch API**：上传一个 JSONL 文件，内含请求列表。承诺 24 小时返回（实际通常约 2-8 小时）。输入和输出 token 各打五折。端点为 `/v1/batches`。符合缓存条件的输入还可以在此基础上叠加缓存输入定价。

**Anthropic Message Batches**：JSONL 上传。24 小时返回。打五折。支持 `cache_control`——缓存写入需要显式声明，批量内部的缓存读取自动发生。

**Google Vertex AI Batch Prediction**：以 BigQuery 或 GCS 作为输入。Gemini 同样享受五折优惠。与 Vertex 流水线集成。

### 语义：是异步，不是慢

批量的含义是"我承诺 24 小时内返回"——而不是"这要花 24 小时"。典型 P50 在 2-6 小时。厂商会把你的批量任务调度到 GPU 库存利用率较低的非高峰时段执行。

### 与缓存叠加

一个 5 万份文档的摘要任务，所有调用共享同一段 4K token 的系统提示词：

- 同步无缓存：50000 × ($input × 4000 + $output × 200)，全价计费。
- 同步带缓存：系统提示词在首次写入后被缓存；其余 49999 次调用的输入便宜 10 倍。
- 批量带缓存：在上述基础上，缓存读和缓存写再各打五折。

叠加效果：批量 + 缓存 ≈ 同步无缓存账单的 10% 左右。任何过夜运行、且有共享系统提示词的工作负载，都应该这样用。

### 工作负载分流

**交互式**——用户在等响应。首 token 延迟（TTFT）很关键。同步调用加提示词缓存。不能走批量。

**半交互式**——用户提交任务后，几分钟后回来查看。异步队列，批量不可用时降级到同步。典型例子是中等规模的 RAG 索引构建。

**批量**——用户期望"第二天早上"或"下个小时"拿到结果。内容流水线、大规模分类、离线分析。永远走批量，永远叠加缓存。

常见错误：因为流水线是"生产环境"就把一切归为交互式。生产环境不是延迟规格——SLA 才是。

### 部分交互性陷阱

有些功能看起来是交互式的，实际能容忍 5-10 分钟。例如：带"刷新"按钮的夜间客户健康度报告。用户点了刷新，等 10 分钟完全没问题。团队却按同步方式上线。50 个并发刷新的成本，是"批量处理后通过邮件送达"方案的 10 倍。

该问的问题是："对这个用户来说，24 小时意味着什么？"如果答案是"他们根本不会注意到"，那就走批量。

### 输出格式陷阱

批量文件格式因厂商而异：

- OpenAI：JSONL，每行一个请求。
- Anthropic：JSONL，每行一条消息；响应格式内嵌其中。
- Vertex：BigQuery 表，或带 TFRecord 的 GCS 前缀。

想跨厂商写"一个统一批量客户端"，意味着要为每家厂商写适配器代码。那些宣称支持多厂商批量的网关（Portkey、LiteLLM 的部分档位），实际也只是对原始格式做了一层薄封装。

### 你应该记住的数字

- 各厂商的批量折扣：输入 + 输出统一打五折。
- 返回时间 SLA：保证 24 小时，典型 P50 为 2-6 小时。
- 批量 + 缓存输入叠加：约为同步无缓存成本的 10%。
- 工作负载分流规则：只要 24 小时延迟可接受，永远走批量。

## 练习

1. 运行 `code/main.py`。对一条 10 万份文档的流水线（3K token 系统提示词、500 token 输出），计算全套叠加方案（批量 + 缓存）相对同步基线的节省额。
2. 在你熟悉的真实产品里挑三个功能，把每个功能分流到交互式/半交互式/批量。
3. 有用户抱怨他的报告花了 3 个小时。这是批量分流失误，还是合理的交互式需求？写出判定标准。
4. 你的批量 API 返回 SLA 是 24 小时，但 P99 是 20 小时。你如何向用户传达这一点——在这种边界情况下，下游系统应该如何处理？
5. 计算盈亏平衡点：共享前缀长到多少 token 时，批量 + 缓存会比在自己预留的 GPU 上过夜运行更便宜？

## 关键术语

| 术语 | 大家怎么说 | 实际含义 |
|------|----------------|------------------------|
| Batch API | "异步折扣" | 五折优惠，24 小时返回 |
| JSONL | "批量格式" | 每行一个 JSON 请求；OpenAI/Anthropic 的标准格式 |
| Message Batches | "Anthropic 批量" | Anthropic 批量 API 的产品名 |
| Batch prediction | "Vertex 批量" | Vertex AI 的批量 API 产品 |
| 返回时间 SLA | "24 小时承诺" | 是保证上限，不是典型值；典型值为 2-6 小时 |
| 工作负载分流 | "交互性决策" | 交互式 / 半交互式 / 批量的路由决策 |
| 输出格式 | "响应格式" | 各厂商的 JSONL 布局；互不通用 |
| 叠加折扣 | "批量 + 缓存" | 两者都生效时约为无缓存同步账单的 10% |

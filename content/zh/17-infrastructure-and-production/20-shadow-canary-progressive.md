# LLM 的影子流量、金丝雀发布与渐进式部署

> LLM 上线集齐了软件部署中最难的几个要素：没有单元测试、故障模式分散、信号反馈滞后。标准流程是：(1) 影子模式（shadow mode）——把生产请求复制一份发给候选模型，只记录日志并做对比，对用户零影响；能发现明显的分布问题，但不构成质量保证；(2) 金丝雀发布（canary rollout）——按 10% → 25% → 50% → 75% → 100% 渐进切换流量，每一步设置门禁；跟踪延迟分位数、单请求成本、错误/拒答率、输出长度分布、用户反馈率；(3) 在稳定性确认后，对差异明显的备选方案做 A/B 测试。非确定性无法消除——由于 GPU 浮点运算不满足结合律加上批大小差异，相同输入在多次运行之间的准确率波动可达 15%。成本是变量而非常量——效果好 20% 的模型，单次调用可能贵 3 倍。回滚速度是决定性因素：如果回滚需要重新部署，那就太慢了。策略放在配置/特性开关里；模型放在注册表里并固定摘要（digest）；回滚 = 翻转策略 + 还原阈值 + 固定回旧模型，几秒钟完成。

**Type:** Learn
**Languages:** Python (stdlib, toy canary-progression simulator)
**Prerequisites:** Phase 17 · 13 (Observability), Phase 17 · 21 (A/B Testing)
**Time:** ~60 minutes

## 学习目标

- 区分影子模式（零影响对比）、金丝雀（真实流量渐进切换）和 A/B 测试（稳定性确认后的对比）。
- 列举五项 LLM 专属的金丝雀指标（延迟、单请求成本、错误/拒答、输出长度分布、用户反馈）。
- 解释为什么 LLM 的非确定性（最高 15%）会改变发布过程中"稳定"的定义。
- 设计一条秒级（策略翻转）而非小时级（重新部署）的回滚路径。

## 问题背景

你上线了一个新模型。离线评测显示准确率提升 3%。你直接在生产环境切换上去。24 小时内，成本上涨 40%，用户点踩率上升 8%，三个客户工单反馈"回答很奇怪"。你决定回滚。重新部署花了 3 小时。你的周末毁了。

这一切本来都可以避免。影子模式会在任何用户看到之前就发现 40% 的成本飙升。金丝雀会在点踩率出现波动时停在 10% 流量。基于策略开关的回滚只需要 30 秒。这套纪律正是用来填补"离线评测看着不错"与"真实用户满意"之间的鸿沟的。

## 核心概念

### 影子模式

候选模型接收与生产环境完全相同的请求；输出只记录日志，不返回给用户。对用户零影响。需要记录：

- 输出内容（与生产输出做 diff）。
- token 数量（成本差异）。
- 延迟。
- 拒答和错误。

能发现：成本暴涨、长度回归、明显的拒答行为变化、硬性错误。不能发现：用户能感知到的质量差异。影子模式是冒烟测试，不是质量测试。

### 金丝雀发布

带门禁的渐进式流量切换。典型节奏：1% → 10% → 25% → 50% → 75% → 100%。每一步对 5 项指标设门禁：

1. **延迟分位数** —— P50、P95、P99。触线条件：金丝雀的 P99 > 基线的 1.5 倍。
2. **单请求成本** —— 综合美元成本。触线条件：高于基线 20% 以上。
3. **错误/拒答率** —— 5xx 加显式拒答。触线条件：达到基线的 2 倍。
4. **输出长度分布** —— 均值 + P99。触线条件：分布发生偏移。
5. **用户反馈率** —— 点踩 / 工单提交。触线条件：达到基线的 1.5 倍。

### 非确定性是新的方差来源

相同的输入会产生不相同的输出。原因包括：

- GPU 浮点运算不满足结合律（浮点归约顺序随批次变化）。
- 批大小差异（同一条 prompt 放在 128 的批里和 16 的批里结果不同）。
- 采样（temperature > 0）。

实测：在完全相同的评测集上，多次运行之间的准确率波动可达 15%。发布过程中的"稳定"指的是指标落在预期方差范围内，而不是与基线完全一致。门禁阈值要设在噪声底线之上。

### 成本是变量

效果好 20% 的模型，单次调用可能贵 3 倍。单请求成本是五项门禁之一。上线一个"更好"但破坏单位经济模型的模型，属于应当回滚的情况。

### 回滚是你的武器

- 策略开关（特性开关系统）：在配置里翻转流量百分比；几秒钟完成。
- 模型固定（注册表摘要）：被固定的模型不会自动升级。
- 回滚 = 还原开关 + 把固定摘要设回上一个版本。秒级，而非小时级。

如果你的技术栈回滚必须重新部署，先修好这个问题再发布。

### 工具

**Argo Rollouts** / **Flagger** —— Kubernetes 渐进式交付控制器。与 Istio/Linkerd 的加权路由集成。

**Istio 加权路由** —— 服务网格层面的流量切分。

**KServe / Seldon Core** —— 内置金丝雀能力的模型服务框架。

**特性开关** —— LaunchDarkly、Flagsmith、Unleash。策略层翻转，无需重新部署。

### 指标检查节奏

金丝雀门禁每 5-15 分钟检查一次，具体取决于流量大小。1% 流量在每分钟 10 个请求的情况下，每个窗口只有 50-150 个数据点——对延迟来说够用，但对用户反馈来说噪声很大。10% 流量大约多 10 倍。每个阶段的停留时间要足够长，以便积累足够的样本。

### A/B 这一步是可选的

如果新模型差异明显（行为不同、成本曲线不同、语气不同），在金丝雀通过后以 50% 流量做 A/B 测试。如果只是同一模型的改进版本，金丝雀门禁通过后直接切到 100%。

### 你应该记住的数字

- 金丝雀节奏：1% → 10% → 25% → 50% → 75% → 100%。
- 非确定性上限：相同输入在多次运行之间的波动可达 15%。
- 五项金丝雀指标：延迟、成本、错误/拒答、输出长度、用户反馈。
- 成本门禁：高于基线 20% 即触线。
- 回滚：秒级，而非小时级。

## 关键术语

| 术语 | 人们怎么说 | 实际含义 |
|------|----------------|------------------------|
| 影子模式（Shadow mode） | "复制流量给新模型" | 零影响地把请求发给候选模型并记录日志 |
| 金丝雀（Canary） | "渐进流量" | 带门禁的、逐步向用户暴露的发布 |
| 门禁（Gates） | "发布检查" | 阻止流量推进的指标阈值 |
| 非确定性（Non-determinism） | "LLM 方差" | 无法消除的多次运行间差异 |
| 策略开关（Policy flag） | "翻转开关回滚" | 配置层回滚，秒级而非小时级 |
| 模型固定（Model pin） | "注册表摘要" | 指向某个模型版本的不可变引用 |
| Argo Rollouts | "K8s 渐进式发布" | Kubernetes 原生的金丝雀/回滚控制器 |
| KServe | "K8s 推理" | 自带金丝雀原语的模型服务框架 |
| Istio 加权路由（Istio weighted） | "网格切流" | 服务网格流量切分器 |

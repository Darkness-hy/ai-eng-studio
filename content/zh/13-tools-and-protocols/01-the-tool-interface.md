# 工具接口 —— 为什么智能体需要结构化 I/O

> 语言模型产出 token，程序执行动作。连接这两者的桥梁就是工具接口（tool interface）：一份让模型请求动作、由宿主程序实际执行的契约。2026 年的每一套技术栈 —— OpenAI、Anthropic、Gemini 的函数调用（function calling）、MCP 的 `tools/call`、A2A 的任务分块 —— 都只是同一个四步循环的不同编码方式。本课为这个循环命名，并展示运行它所需的最小机制。

**Type:** Learn
**Languages:** Python (stdlib, no LLM)
**Prerequisites:** Phase 11 (LLM completion APIs)
**Time:** ~45 minutes

## 学习目标

- 解释为什么一个只能生成文本的 LLM 无法独立对真实世界采取行动。
- 画出工具调用的四步循环（描述 → 决策 → 执行 → 观察），并说出每一步由谁负责。
- 把一个工具描述写成三部分：名称、JSON Schema 输入、以及一个确定性的执行器函数。
- 区分纯函数型工具与有副作用的工具，并说明这一区分对安全为何重要。

## 问题背景

LLM 输出的是下一个 token 的概率分布，这就是它全部的输出能力。如果你问一个聊天模型"班加罗尔现在的天气怎么样"，它能写出一句听上去合理的话，但它无法接入天气 API。这句话可能恰好蒙对，也可能是三天前的旧信息。

工具接口的目的就是填补这个鸿沟。宿主程序 —— 你的智能体运行时、Claude Desktop、ChatGPT、Cursor，或者一个自定义脚本 —— 向模型公布一份可调用的工具清单。当模型判断需要执行某个动作时，它会输出一个结构化载荷，指明工具名称及其参数。宿主解析该载荷，真正运行这个工具，并把结果回传给模型。循环持续进行，直到模型判断不再需要更多调用为止。

这份契约的第一个版本于 2023 年 6 月以 OpenAI 的 "functions" 参数形式发布。Anthropic 随后在 Claude 2.1 中推出了 `tool_use` 块。几个月后 Gemini 加入了 `functionDeclarations`。现在所有提供商都暴露同一种形态：输入是一份以 JSON Schema 定义类型的工具清单，输出是一个 JSON 载荷形式的工具调用。Model Context Protocol（2024 年 11 月）把这份契约泛化，让一个工具注册表可以服务于所有模型。A2A（2026 年 4 月，v1.0）又在同一原语之上叠加了智能体间委托的能力。

四步循环是所有这些方案底下不变的核心。第 13 阶段的其余内容都只是对它的展开。

## 核心概念

### 第一步：描述

宿主用三个字段声明每个工具。

- **名称（Name）。** 一个稳定、机器可读的标识符。是 `get_weather`，而不是"那个天气功能"。
- **描述（Description）。** 一段自然语言简介。"当用户询问某个具体城市的当前天气状况时使用。不要用于历史数据。"
- **输入 schema。** 一个描述工具参数的 JSON Schema 对象（draft 2020-12）。

模型会收到这份清单。现代提供商会用各自的模板把这些声明序列化进系统提示词，所以作为调用方，你只需要处理结构化的形式。

### 第二步：决策

给定用户消息和可用工具，模型会从三种行为中选择其一。

1. **直接用文本回答。** 不发起工具调用。
2. **调用一个或多个工具。** 输出结构化的调用对象。在 `parallel_tool_calls: true` 下（OpenAI 和 Gemini 默认开启，Anthropic 需要手动开启），模型可以在一轮中输出多个调用。
3. **拒绝。** 严格模式（strict mode）的结构化输出可以产生一个带类型的 `refusal` 块，而不是一个调用。

工具调用载荷有三个稳定字段：调用 `id`、工具 `name`，以及一个 JSON 格式的 `arguments` 对象。id 的存在是为了让宿主能把后续的结果与具体的某次调用对应起来 —— 当并行调用乱序返回时这一点尤为重要。

### 第三步：执行

宿主接收调用，依照声明的 schema 校验参数，然后运行执行器。参数无效意味着模型幻觉出了某个字段或用错了类型 —— 这是弱模型上非常常见的失败模式。面对无效参数，生产环境的宿主通常采取三种做法之一：快速失败并把错误反馈给模型、用受约束的解析器修复 JSON、或者把校验错误附在提示词里重试模型。

执行器本身就是普通代码。Python、TypeScript、一条 shell 命令、一次数据库查询。它产出一个结果，通常是字符串，但也可以是任意 JSON 值或一个结构化内容块（MCP 中的文本、图片或资源引用）。结果必须是可序列化的。

### 第四步：观察

宿主把工具结果追加到对话中（作为一条带匹配 `id` 的 `tool` 角色消息），然后再次调用模型。此时模型的上下文中已经有了工具输出，它可以给出最终答案，也可以请求更多调用。这个过程持续到模型不再输出调用，或者宿主触发迭代次数的安全上限为止。

### 信任划分

从安全角度看，工具分为两类。

- **纯函数型（Pure）。** 只读、确定性、无副作用。`get_weather`、`search_docs`、`get_current_time`。可以放心地投机式调用。
- **有后果型（Consequential）。** 会改变状态、花钱、触碰用户数据。`send_email`、`delete_file`、`execute_trade`。必须设置门禁。

Meta 在 2026 年提出的智能体安全"二选规则"（Rule of Two）指出：单轮交互最多只能同时组合以下三项中的两项 —— 不可信输入、敏感数据、有后果的动作。而工具接口正是落实这条规则的地方 —— 通过拒绝调用、要求用户确认或升级权限范围。完整的安全章节见第 13 阶段第 15 课，智能体级权限策略见第 14 阶段第 09 课。

### 循环存在于何处

| 场景 | 谁负责描述 | 谁负责决策 | 谁负责执行 |
|---------|---------------|-------------|--------------|
| 单轮函数调用（OpenAI/Anthropic/Gemini） | 应用开发者 | LLM | 应用开发者 |
| MCP | MCP 服务器 | LLM（经由 MCP 客户端） | MCP 服务器 |
| A2A | Agent Card 发布者 | 调用方智能体 | 被调用方智能体 |
| 网页浏览器（函数调用型智能体） | 浏览器扩展 / WebMCP | LLM | 浏览器运行时 |

在所有场景下，都是同样的四个步骤。列名在变，结构不变。

### 为什么不直接提示模型输出 JSON？

"让模型用 JSON 回复"是函数调用出现之前的常见做法。在前沿模型上它的失败率约为 5% 到 15%，在小模型上则高得多。失败模式包括括号缺失、多余的逗号、幻觉出来的字段、错误的类型。然后你就得加一道 JSON 修复、重试，或者上受约束解码器。

原生函数调用胜出的原因有三。第一，提供商针对精确的调用格式对模型做端到端训练，严格模式下有效 JSON 率攀升到 98% 到 99%。第二，调用载荷位于独立的协议槽位中，不混在自由文本里 —— 因此工具调用永远不会泄漏到用户可见的回复中。第三，提供商用受约束解码（constrained decoding）强制 schema 合规（OpenAI 的 strict mode、Anthropic 的 `tool_use`、Gemini 的 `responseSchema`）。输出保证能通过校验。

第 13 阶段第 02 课会并排讲解三家提供商的 API。第 13 阶段第 04 课深入讨论结构化输出。

### 熔断机制

循环在模型停止输出调用或宿主达到最大轮数时终止。生产环境的宿主通常把这个上限设在 5 到 20 轮之间。超过这个数，你几乎肯定陷入了一个模型无法自行跳出的循环。Claude Code 默认为 20，OpenAI Assistants 为 10，Cursor 的智能体模式为 25。

另一种选择 —— 无上限循环 —— 每隔半年就会以"智能体一夜之间烧掉 400 美元 API 费用"的事故复盘形式出现一次。不要在没有上限的情况下上线。

## 生产实践

`code/main.py` 在没有 LLM 的情况下运行这个四步循环。一个假的"决策器"函数通过对用户消息做模式匹配来模拟模型；而执行器、schema 校验器和观察步骤的运行框架都是真实的。运行它，观察完整的请求/响应编排过程及可打印的中间状态，之后在后续课程中把假决策器替换成任意真实提供商即可。

值得关注的内容：

- 工具注册表为每个工具保存三个字段：名称、描述、schema，外加一个执行器引用。
- 校验器是一个只用标准库实现的最小 JSON Schema 子集（类型、required、enum、min/max）。第 13 阶段第 04 课会提供一个更完整的版本。
- 循环把迭代次数限制在五次。生产环境的智能体恰恰需要这种熔断机制。

## 交付产物

本课产出 `outputs/skill-tool-interface-reviewer.md`。给定一份工具定义草稿（名称 + 描述 + schema + 执行器概要），该 skill 会审查它是否适配这个循环：名称是否机器稳定、描述是否构成完整的使用说明、schema 是否正确使用了 JSON Schema 2020-12、纯函数型与有后果型的分类是否明确标注。

## 练习

1. 给 `code/main.py` 添加第四个工具 `get_stock_price(ticker)`。把它的描述写成 "Use when the user asks for a current stock price by ticker. Do not use for historical prices or market summaries."。运行框架，确认假决策器会把提到股票代码的查询路由到新工具。

2. 故意破坏 schema 校验器。传入一个 `arguments` 对象缺少必填字段的调用，确认宿主在执行前就拒绝了它。然后传入一个带额外未知字段的调用。做出决定：宿主应该拒绝还是忽略？用一条安全论证来支撑你的选择。

3. 把框架中的每个工具分类为纯函数型或有后果型。给需要的注册表条目添加 `consequential: true` 标记，并修改循环，使其在选中有后果型工具时打印一行 "would confirm with user"。这就是每个生产宿主都需要的确认门禁的雏形。

4. 在纸上画出四步循环，并针对你最常用的客户端（Claude Desktop、Cursor、ChatGPT 或自定义技术栈）填好上文的提供商对照表。与第 13 阶段第 06 课中 MCP 特定的变体相互对照。

5. 从头到尾读一遍 OpenAI 的函数调用指南。找出请求中存在、但不在本课呈现的四步循环里的那一个字段。解释它带来了什么，以及为什么它属于便利项而非必需项。

## 关键术语

| 术语 | 大家口中的说法 | 实际含义 |
|------|----------------|------------------------|
| 工具（Tool） | "模型能调用的东西" | 名称 + JSON Schema 类型化输入 + 执行器函数的三元组 |
| 函数调用（Function calling） | "原生工具使用" | 提供商在 API 层面支持输出结构化工具调用，而非散文文本 |
| 工具调用（Tool call） | "模型发出的行动请求" | 模型输出的、含 `id`、`name`、`arguments` 的 JSON 载荷 |
| 工具结果（Tool result） | "工具返回的东西" | 执行器的输出，包装在带匹配 id 的 `tool` 角色消息中 |
| 并行工具调用（Parallel tool calls） | "一次发多个调用" | 单轮模型输出中的多个调用对象，彼此独立、可按 id 排序 |
| 严格模式（Strict mode） | "保证合法的 JSON" | 受约束解码，强制模型输出通过所声明 schema 的校验 |
| 纯函数型工具（Pure tool） | "只读工具" | 无副作用，可以安全地重复运行 |
| 有后果型工具（Consequential tool） | "动作工具" | 会改变外部状态，需要门禁、审计或用户确认 |
| 四步循环（Four-step loop） | "工具调用周期" | 描述 → 决策 → 执行 → 观察 |
| 宿主（Host） | "智能体运行时" | 持有工具注册表、调用模型并运行执行器的程序 |

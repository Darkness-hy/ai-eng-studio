# Qwen-VL 家族与动态 FPS 视频处理

> Qwen-VL 家族——Qwen-VL（2023）、Qwen2-VL（2024）、Qwen2.5-VL（2025）、Qwen3-VL（2025）——是 2026 年最具影响力的开源视觉-语言模型谱系。每一代都押下了一个决定性的架构赌注，并在十二个月内被整个开源生态争相效仿：基于 M-RoPE 的原生动态分辨率、带绝对时间对齐的动态 FPS 采样、ViT 中的窗口注意力，以及结构化的智能体输出格式。到 Qwen3-VL 时，这套配方已经稳定下来：采用原生宽高比输入的 2D-RoPE-ViT 编码器、接入大型 Qwen3 语言底座的 MLP 投影器，以及把 OCR、定位（grounding）和智能体行为当作一等目标的训练阶段。本课按时间顺序读完整个家族，让你理解每个设计细节为什么会出现在它现在的位置上。

**Type:** Learn
**Languages:** Python (stdlib, M-RoPE encoder + dynamic-FPS sampler)
**Prerequisites:** Phase 12 · 06 (patch-n'-pack)
**Time:** ~120 minutes

## 学习目标

- 计算 M-RoPE 的三轴旋转（时间、高度、宽度），并解释为什么三个轴缺一不可。
- 为一段视频选择动态 FPS 采样策略，并权衡每秒 token 数与事件检测准确率之间的关系。
- 按顺序说出 Qwen-VL 四代的升级要点，以及每次升级带来了什么能力。
- 搭建 Qwen2.5-VL 风格的 JSON 智能体输出格式，并从 VLM 的响应中解析结构化工具调用。

## 问题背景

Qwen-VL 于 2023 年 8 月发布，直接对标 LLaVA-1.5 和 BLIP-2。Qwen 团队瞄准的差距有三个方面：分辨率、视频和结构化输出。

分辨率：LLaVA-1.5 运行在 336x336。看照片够用，但面对中文发票或密密麻麻的电子表格截图就毫无用处。Qwen-VL 的第一项创新是 448x448 分辨率加上带定位的边界框输出，让模型能够「指出」目标。

视频：Video-LLaMA 把逐帧编码器堆叠起来再喂给 LLM。短视频片段还能凑合，但对于以时间轴为核心信号的多分钟视频就不行了。Qwen 团队想要的是一个能理解时间的单一编码器。

结构化输出：LLaVA 只会输出自由格式文本。智能体需要的是 JSON。Qwen-VL 在训练时使用了显式的 JSON 输出格式，包括以文本形式表示的边界框坐标。

之后的每一代 Qwen-VL 都在这三条轴线之一上做了延伸。

## 核心概念

### Qwen-VL（2023 年 8 月）

第一代：OpenCLIP ViT-bigG/14 作为编码器（25 亿参数），兼容 LLama 的 Q-Former（单步、256 个查询向量），Qwen-7B 底座。主要贡献：

- 448x448 分辨率（当时开源 VLM 的 SOTA）。
- 定位能力：在带显式坐标 token 输出的图文对上训练。例如 "The cat is at <box>(112, 204), (280, 344)</box>"。
- 从一开始就进行中英双语训练。

当时的基准成绩：英文任务上与 GPT-4V 不相上下，中文任务上占据优势。真正的亮点是定位监督。

### Qwen2-VL（2024 年 9 月）—— M-RoPE 与原生分辨率

Qwen2-VL 用原生动态分辨率的 ViT 编码器取代了「固定分辨率 + Q-Former」的组合。关键变化：

- 原生动态分辨率。ViT 接受任意能被 28 整除的 HxW（patch 大小 14，配合 2x 空间合并）。一张 1120x672 的图像（合并后 40x24 个 patch）产生 960 个视觉 token。不缩放、不切片、不做缩略图。
- M-RoPE（多模态 RoPE）。每个 token 携带 3D 位置 (t, h, w) 而非 1D。图像 t=0，视频 t = frame_index。RoPE 按每个轴各自的频率旋转 query/key 向量。不需要位置嵌入表。
- MLP 投影器。抛弃 Q-Former；在合并后的 patch token 上使用一个 2 层 MLP。
- 支持动态 FPS 的视频。视频默认按 1-2 FPS 采样，但模型可以接受任意帧数。

结果：Qwen2-VL-7B 在多个多模态基准上追平 GPT-4o，并在 DocVQA 上超越它（94.5 对 88.4）。架构变革是这次的决定性一步。

### Qwen2.5-VL（2025 年 2 月）—— 动态 FPS + 绝对时间

Qwen2.5-VL 的重大转变在于视频。动态 FPS 不只是「需要时多采几帧」。论文将其形式化为：

- 绝对时间 token。不再使用位置索引（第 0、1、2 帧……），而是使用真实时间戳。「在 0:04 时，猫跳了起来。」模型看到的是与帧 token 交错排列的 `<time>0.04</time>` token。
- 动态 FPS。慢节奏画面按 1 FPS 采样，动作画面按 4+ FPS 采样。由用户或训练者选择；M-RoPE 自动适配。
- ViT 中的窗口注意力。为了吞吐量，空间注意力被限制在窗口内（块内局部）；每隔几层加入一次全局注意力。
- 显式 JSON 输出格式。在工具调用数据上训练："{\"tool\": \"click\", \"coords\": [380, 220]}"。开箱即可用于智能体。
- MRoPE-v2 缩放。位置编码随最大输入尺寸缩放，使 10 分钟的视频不会超出频率范围。

基准成绩：Qwen2.5-VL-72B 在大多数视频基准上击败 GPT-4o，在文档任务上追平 Gemini 2.0，并创下开源模型在 GUI 定位上的 SOTA（ScreenSpot：84% 准确率，GPT-4o 为 38%）。

### Qwen3-VL（2025 年 11 月）

Qwen3-VL 是一次巩固而非重塑的增量升级：更大的 LLM 主干（Qwen3-72B）、扩充的训练数据、改进的 OCR，以及借助 Qwen3「思考模式」获得的更强推理能力。ViT 和 M-RoPE 保持不变。论文的重点是数据与训练改进，而非架构。

这个谱系的启示：到 2025 年，Qwen-VL 架构已经稳定。后续世代扩展的是算力和数据，而不是基础组件。

### M-RoPE 的数学表达

经典 RoPE 对维度为 `d` 的 query `q` 按位置 `m` 进行成对坐标旋转：

```
q_rot[2i]   = q[2i]   * cos(m * theta_i) - q[2i+1] * sin(m * theta_i)
q_rot[2i+1] = q[2i]   * sin(m * theta_i) + q[2i+1] * cos(m * theta_i)
theta_i     = 10000^(-2i/d)
```

M-RoPE 把隐藏维度切成三个频带。假设 `d = 96`，分配 32 维给时间轴、32 维给高度轴、32 维给宽度轴。每个频带按各自轴的位置进行旋转。一个位于 (t=5, h=10, w=20) 的 patch 会在三个频带上分别应用旋转 `R_t(5)`、`R_h(10)`、`R_w(20)`。

文本 token 使用 `t = text_index, h = 0, w = 0`（或某种归一化方案），从而保持兼容。视频帧使用 `t = frame_time, h = row, w = col`。单张图像使用 `t = 0`。

好处在于：一套位置编码同时处理文本、图像和视频，不需要分支代码，也不需要多套位置表。

### 动态 FPS 采样逻辑

给定一段时长为 `T` 秒的视频和目标 token 预算 `B`：

1. 计算能承受的最大 FPS：`fps_max = B / (T * tokens_per_frame)`。
2. 从 `{1, 2, 4, 8}` 中选出满足 `fps <= fps_max` 的目标 FPS。
3. 如果画面运动剧烈（基于光流启发式或用户显式要求），选更高的 FPS；运动平缓则选更低的。
4. 按选定 FPS 均匀采样；在帧之间插入 `<time>t</time>` token。

Qwen2.5-VL 通过训练隐式学会了这套逻辑；推理时由用户通过 `fps` 参数控制。一段 60 秒的动作片段，按 4 FPS、每帧 81 个 token 计算，共 19440 个 token，在 32k 上下文中绰绰有余。

### 结构化智能体输出

Qwen2.5-VL 的智能体训练明确以结构化工具调用为目标：

```
{
  "tool": "mouse_click",
  "coords": [1024, 512],
  "button": "left",
  "modifier": null
}
```

解析是确定性的：对模型输出直接 JSON.parse。对比自由格式的 "click at (1024, 512)"，后者需要正则表达式和歧义处理。正是这一转变让 Qwen2.5-VL 的 ScreenSpot 分数从 Qwen2-VL 的 55% 跃升到 84%。

## 生产实践

`code/main.py` 实现了：

- 为混合了文本、图像 patch 和视频帧的打包序列计算 M-RoPE 位置。
- 动态 FPS 采样器：给定 (duration, budget, motion_level)，选择 FPS 并输出帧时间戳。
- 一个玩具版的 Qwen2.5-VL JSON 输出解析器，处理带坐标字段的工具调用响应。

运行它，然后在一段 5 分钟的视频上把固定 FPS 换成动态 FPS，亲自感受其中的差别。

## 练习

1. 为位于 (t=3, h=5, w=7) 的 patch 计算 M-RoPE 旋转，隐藏维度为 48（每个频带 16 维，基础 theta 为 10000）。写出每个频带前三对坐标的旋转角度。

2. 一段 10 分钟的监控录像按 1 FPS 采样会产生多少帧？在 384 分辨率、3x 池化下，总共有多少个 token？Qwen2.5-VL 默认的 32k 上下文能装下吗？

3. 分别为 30 秒的网球对拉、30 秒的菜谱演示和 30 秒的 UI 智能体录屏选择 FPS。用动态 FPS 的逻辑为每个选择给出理由。

4. Qwen2.5-VL 完全抛弃了 Q-Former。为什么一个简单的 MLP 在 2025 年可行，而在 2023 年不行？（提示：数据规模和编码器质量。）

5. 把三条 Qwen2.5-VL 的 JSON 工具调用输出解析成 Python 字典。遇到格式错误的 JSON 时会出什么问题？Qwen cookbook 推荐什么恢复策略？

## 关键术语

| 术语 | 常见说法 | 实际含义 |
|------|-----------------|------------------------|
| M-RoPE | 「多模态 RoPE」 | 在隐藏维度中划分时间、高度、宽度三个频带的 3D 旋转位置编码 |
| 动态 FPS | 「智能采样」 | 根据运动幅度、时长和 token 预算为每段视频单独选择的帧采样率 |
| 绝对时间 token | 「时间戳 token」 | 交错插入序列中的 `<time>t</time>`，让模型看到真实秒数而非帧索引 |
| 窗口注意力 | 「局部注意力」 | 为提速而限制在小窗口内的空间自注意力；周期性地补充全局注意力 |
| 结构化智能体输出 | 「JSON 模式」 | 通过训练数据监督，让 VLM 输出带坐标和工具名的可解析 JSON |
| min_pixels / max_pixels | 「分辨率上下限」 | Qwen2.5-VL 的按请求控制项，约束总像素数从而约束 token 数 |
| 定位（Grounding） | 「指出目标」 | 以文本 token 形式输出边界框坐标；从 Qwen-VL 第一代沿用至今 |

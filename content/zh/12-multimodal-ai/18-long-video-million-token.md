# 百万 Token 上下文下的长视频理解

> 一段 1 小时的 4K 视频在 24 FPS 下经过分块和嵌入处理，会产生约 6000 万个 token。一期 2 小时的播客转录成文字是 3 万个 token。一部完整的蓝光电影，即便用激进的池化压缩，也有数十万个 token。Google 的 Gemini 1.5（2024 年 3 月）以 1000 万 token 的上下文开启了这个时代，能在长达一小时的视频上做可靠的"大海捞针"式召回。LWM（Liu et al.，2024 年 2 月）展示了环形注意力（ring attention）的扩展路径。LongVILA 和 Video-XL 进一步扩大了视频摄入规模。VideoAgent 则用智能体检索取代了原始上下文。每种方法都是在计算量、召回率和工程复杂度之间的不同权衡。本课把它们放在一起对照阅读。

**Type:** Build
**Languages:** Python (stdlib, needle-in-haystack simulator + agentic-retrieval router)
**Prerequisites:** Phase 12 · 17 (video temporal tokens)
**Time:** ~180 minutes

## 学习目标

- 计算长视频在不同 FPS 和池化设置下的视觉 token 总数。
- 解释三条扩展路径：暴力上下文（Gemini 1.5）、环形注意力（LWM）、token 压缩（LongVILA / Video-XL）。
- 在准确率和延迟两个维度上，对比原始上下文视频 VLM 与智能体检索视频 VLM（VideoAgent）。
- 为一段 30 分钟的视频设计"大海捞针"测试，并测量特定分钟处的召回率。

## 问题背景

一帧 Qwen2.5-VL 规格的图像块在 384 原生分辨率下约为 729 个 token。经过 3x3 池化后每帧 81 个 token。一段 30 分钟的视频以 1 FPS 采样 = 1800 帧 = 145,800 个 token。2025 年的开源 VLM 能处理，但很紧张。换成 2 FPS 就是 291,600 个 token——只有最大的上下文才装得下。

一部 2 小时的电影以 1 FPS 采样是 58.3 万个 token。超出了大多数 2026 年开源模型的能力；需要 Gemini 2.5 Pro，或者更激进的池化。

由此出现了三条扩展路径。

## 核心概念

### 路径 1：暴力上下文（Gemini 1.5、Claude Opus）

用硬件砸钱解决问题。把上下文扩展到数百万 token，在一次前向传播中处理全部内容。

Gemini 1.5 Pro 发布时支持 100 万 token；Gemini 1.5 Ultra 达到 1000 万；2026 年的 Gemini 2.5 Pro 可以可靠地处理数小时的视频。论文（arXiv:2403.05530）记录了在最高约 950 万 token 时"大海捞针"召回率达 99.7%。

工程实现：一套带有内存层级（局部 + 全局 + 稀疏）的定制注意力实现，加上面向长上下文效率的 MoE 专家路由。完整细节未公开。不开源。

### 路径 2：环形注意力（LWM、LongVILA）

环形注意力把长序列分布到一个"环"上的多个设备中，每个设备持有一个分块。整个序列上的注意力计算是这样完成的：每个设备按环形模式把自己的分块发送给下一个设备，计算部分注意力，再做聚合。

LWM（Liu et al.，2024）用这种方式训练了一个 100 万 token 上下文的模型。训练计算量随上下文线性增长，而不是平方增长——注意力的平方开销被摊销到环上的各个设备上。

LongVILA（arXiv:2408.10188）把这个模式移植到了 VLM。1400 帧视频、每帧 192 个 token = 26.8 万上下文，用 8 路并行的环形注意力训练。

### 路径 3：token 压缩（Video-XL、LongVA）

比暴力上下文更便宜的方案：在序列进入 LLM 之前先做激进压缩。

Video-XL（arXiv:2409.14485）使用视觉摘要 token：每个由 N 帧组成的片段产生一个对这 N 帧做注意力的"摘要"token。推理时，LLM 每个片段只看到一个摘要 token，大幅缩小了上下文。

LongVA 用一种"长上下文迁移"技术把 LLM 上下文从 20 万扩展到 200 万。在长上下文文本上训练，再通过共享表示迁移到长上下文视频。

token 压缩用特定时间戳上的召回率换取了可扩展性。模型大致知道发生了什么，但有时会漏掉确切的帧。

### 路径 4：智能体检索（VideoAgent）

不把完整视频喂给 LLM。而是把视频当作一个数据库，用 LLM 去查询它。

VideoAgent（arXiv:2403.10517）：

1. LLM 读取问题。
2. LLM 向检索工具请求相关片段（"给我看包含猫的片段"）。
3. 工具返回匹配片段的时间戳。
4. LLM 通过一个 VLM 读取这些片段。
5. LLM 组织答案，或继续追加查询。

这是把 LLM-as-agent 模式应用到长视频上。推理更便宜（只编码相关片段），工程更难（检索质量成为瓶颈）。

### "大海捞针"基准测试

标准的长上下文测试：在视频的随机位置插入一个独特的视觉或文本标记，然后提出一个需要回忆该标记才能回答的问题。

指标：在不同视频长度和标记位置下的 Recall@k。

Gemini 2.5 Pro 在长达 90 分钟的视频上召回率超过 99%。开源 72B 模型（Qwen2.5-VL-72B、InternVL3-78B）在 30 分钟时约为 85-90%，超过 60 分钟后性能下降。

VideoAgent 在 2 小时以上的视频上可以追平甚至超过原始上下文模型，因为只要检索工具够好，就能命中那根"针"。

### 如何选择路径

15 分钟的视频、要求前沿级准确率：开源 72B + 原生上下文通常就够了。选 Qwen2.5-VL-72B。

30 分钟到 1 小时的内容：开源方案选 LongVILA 或 Video-XL；闭源方案选 Gemini 2.5 Pro。质量门槛决定一切——追求前沿水平就用闭源。

2 小时以上的内容：VideoAgent 或类似的检索模式。或者，把视频总结成更小的块，再输入分层摘要。

### 2026 年的生产模式

实践中，生产环境的长视频流水线是混合式的：

1. 对整个视频运行动态 FPS 采样 + 激进池化（得到一个 10 万 token 的全局表示）。
2. 交给一个 72B VLM 生成全局摘要。
3. 如果用户提出细节性问题，以该摘要为索引运行智能体检索。

这种方式结合了用于全局理解的暴力上下文和用于局部细节的检索。

## 生产实践

`code/main.py`：

- 计算从 1 分钟到 3 小时的视频在不同 FPS + 池化设置下的 token 预算。
- 模拟一次"大海捞针"实验：在随机时间戳注入标记，提出问题，评估召回。
- 包含一个智能体检索路由模拟器，挑选特定片段喂给下游 VLM。

## 练习

1. 一场 45 分钟的讲座，1 FPS 采样，每帧 81 个 token。总 token 数是多少？能装进哪些模型的上下文？

2. 设计一个"大海捞针"测试：你在第几分钟注入标记，确切的查询格式是什么？

3. 在一段 1 小时的视频上，对比暴力上下文的 Qwen2.5-VL-72B（8 万上下文）和 VideoAgent（Claude 3.5 + 检索）。谁在召回率上获胜？谁在延迟上获胜？

4. 环形注意力的内存开销随序列长度线性增长，也随设备数量线性增长。解释原因，以及如果去掉环形轮转阶段会出什么问题。

5. 阅读 Gemini 1.5 论文第 5 节关于"大海捞针"的内容。论文在 100 万与 1000 万 token 边界处的召回率上有什么发现？

## 关键术语

| 术语 | 大家怎么说 | 实际含义 |
|------|-----------------|------------------------|
| 暴力上下文（Brute context） | "就是更多 token" | 把 LLM 上下文扩展到数百万 token；一次前向传播处理全部内容 |
| 环形注意力（Ring attention） | "LWM 式并行" | 分布式注意力模式，每个设备持有一个分块并轮转传递 |
| token 压缩（Token compression） | "摘要 token" | 在进入 LLM 之前，用一个可学习的压缩器减少每个片段的 token 数 |
| 大海捞针（Needle-in-haystack） | "NIH 测试" | 在随机位置插入一个独特标记，测试时要求模型回忆它 |
| 智能体检索（Agentic retrieval） | "LLM 作为查询规划器" | LLM 向检索工具请求相关片段，通过 VLM 读取它们，再组织答案 |
| VideoAgent | "视频版检索模式" | 智能体检索的经典设计：问题 -> 工具 -> 片段 -> 答案 |

# 流水线并行与气泡分析

> 张量并行把矩阵乘法切分到多个 rank 上，而流水线并行把模型切分到多个 rank 上，每个 rank 负责一个阶段。微批次（microbatch）依次流过流水线。流水线开始和结束处的空闲时间就是气泡（bubble）；把气泡降到最小，正是这门手艺的全部精髓。

**Type:** Build
**Languages:** Python
**Prerequisites:** Phase 19 Track C lessons 42-49
**Time:** ~90 min

## 学习目标

- 把一个串行模型拆成 N 个阶段，并在 N 个 rank 上模拟前向流水线。
- 用 GPipe 调度（先全部前向填充流水线，再统一反向）调度 M 个微批次通过流水线，并计算气泡占比。
- 将其气泡与 Megatron-LM 和 PipeDream 使用的交错式 1F1B 调度进行对比。
- 论证阶段划分的原则：各阶段计算量相等比参数量相等更重要。

## 问题背景

一个 70B 参数的模型在 fp16 下光参数就需要 140 GB，没有任何消费级 GPU 装得下。ZeRO-3 把参数分片到各个 rank，但每次前向时仍需每个 rank 对完整的层做 allgather，每层付出 log(N) 跳的通信代价。流水线并行（pipeline parallelism）走的是另一条路：把模型切成 N 个阶段，每个 rank 放一个阶段。第 1 层的前向在 rank 0 上完成后，把激活张量交给 rank 1；rank 1 跑第 2 层后交给 rank 2；以此类推。反向则按相反方向流动。由于每个 rank 只持有一个阶段，显存占用线性下降；但计算是串行的，这就引出了气泡问题。

气泡指的是流水线起始处的空闲时间（等待第一个微批次到达最后一个阶段）和结束处的空闲时间（等待最后一个微批次反向排空整条流水线）。在 M 个微批次、N 个阶段的情况下，每个阶段的气泡占比为 (N-1)/(M+N-1)。当 M=8、N=4 时是 27%；当 M=64、N=4 时是 4.5%。微批次越多，气泡越小，这意味着每个微批次的 batch size 要小，而这正是驱动微批次设计的约束条件。

## 核心概念

```mermaid
flowchart LR
  R0[rank 0: stage 0 / layer 0] --> R1[rank 1: stage 1 / layer 1]
  R1 --> R2[rank 2: stage 2 / layer 2]
  R2 --> R3[rank 3: stage 3 / loss]
  R3 -.backward.-> R2
  R2 -.backward.-> R1
  R1 -.backward.-> R0
```

### GPipe 调度

先用全部 M 个微批次的前向填满流水线，再开始任何反向；然后按相反顺序排空反向。每个微批次的激活必须保留到它的反向开始为止，因此显存随 M 线性增长。前向需要 M+N-1 个周期，反向再需要 M+N-1 个周期。每个阶段的有效工作量是 2M 个周期；每个阶段的气泡是 2(N-1) 个周期。在每次前向和反向各占一个时间单位的假设下，气泡占比为 (N-1)/(M+N-1)。把 M 取得远大于 N 就能掩盖气泡。

### 1F1B 调度

交错执行：只要某个微批次的前向到达最后一个阶段，就立即开始它的反向，让它向回流动。该调度让每个阶段交替执行一次前向、一次反向。气泡仍然是 N-1，但激活显存的上界由流水线深度决定，而不是微批次数量。生产环境的流水线都使用 1F1B（Megatron、PipeDream）。本课先实现更简单的 GPipe，1F1B 留作练习。

### 为什么各阶段计算量相等很重要

如果阶段 0 耗时 50 ms 而阶段 1 耗时 100 ms，那么每个周期都被阶段 1 卡住。其他阶段每个周期都要空等 50 ms，等阶段 1 放行。参数量相等是错误的衡量维度：Transformer 的计算量由每层的注意力加 MLP 主导，而嵌入层参数很多、计算却很少。阶段划分应该让每个阶段的 FLOPs 相等，而不是权重数量相等。

### 微批次与批次

一条流水线运行 M 个大小为 B 的微批次。有效 batch size 是 M*B。一次流水线步结束时的梯度，就是这 M*B 个样本合在一起的梯度。气泡占比取决于 M；优化器看到的是 M*B。调节 M 意味着在气泡（M 越大越低）和每微批次显存（对 GPipe 而言 M 越大激活显存越高）之间做权衡。

## 业界生产模式

三种模式让流水线并行足够可靠、能够上线。

**激活检查点（activation checkpointing）与流水线搭配使用。** 在 GPipe 中有 M 个微批次同时在飞行时，激活显存是单个微批次的 M 倍。激活检查点在反向时重算前向，用计算换显存；正是两者的组合让流水线在长序列场景下变得可行。

**阶段均衡靠实测，不靠假设。** 生产团队会先跑一遍性能分析，在目标硬件上实测每层的真实计算量（FLOPs 和墙钟时间），然后据此划分。Megatron-LM 的 `--num-layers-per-stage` 参数接受一个列表，允许在各阶段单层成本不同时使用不均匀的层数。

**send-recv 调度必须避免死锁。** 如果每个阶段都先发送再接收，流水线会在通信链路上死锁。标准解法是交错执行：偶数 rank 的阶段先 send 再 recv，奇数 rank 的阶段先 recv 再 send。本课显式地编排各个 rank，让这个模式清晰可见。

## 生产实践

生产模式：

- **Megatron-LM。** 大规模流水线并行的参考实现。使用 1F1B，并支持张量并行 + 流水线并行 + 数据并行的组合。
- **DeepSpeed Pipeline。** 与 ZeRO 集成；ZeRO-1 + 流水线是最大规模开源模型的常见组合。
- **PyTorch Pipe。** PyTorch 原生的流水线封装，基于 `torch.distributed.pipeline.sync.Pipe` 构建。

## 关键术语

| 术语 | 通俗说法 | 实际含义 |
|------|----------------|------------------------|
| 流水线（Pipeline） | "沿深度方向的模型并行" | 每个 rank 一个阶段，激活逐阶段流动 |
| 气泡（Bubble） | "流水线空闲时间" | 起始和结束处共 (N-1) 步，期间部分阶段无活可干 |
| 微批次（Microbatch） | "批次的一个切片" | 一次前向/反向的基本单元；M 越大气泡越小 |
| GPipe | "先填满再排空" | 全部 M 次前向完成后才开始反向；激活显存高 |
| 1F1B | "交错调度" | 每个阶段一次前向一次反向交替；激活显存有上界 |

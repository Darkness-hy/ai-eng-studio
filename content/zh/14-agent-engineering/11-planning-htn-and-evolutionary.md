# HTN 规划与进化搜索

> 符号规划（symbolic planning）适用于需要可证明正确性的场景，进化代码搜索（evolutionary code search）适用于适应度函数可由机器自动校验的场景。ChatHTN（2025）和 AlphaEvolve（2025）展示了这两种方法与 LLM 结合后各自能解锁的能力。

**Type:** Build
**Languages:** Python (stdlib)
**Prerequisites:** Phase 14 · 02 (ReWOO and Plan-and-Execute)
**Time:** ~75 minutes

## 学习目标

- 解释分层任务网络（Hierarchical Task Network, HTN）：任务、方法、算子、前置条件、效果。
- 描述 ChatHTN 的混合循环——符号搜索加 LLM 兜底分解。
- 解释 AlphaEvolve 的进化循环，以及它为什么只在评估器可编程化时才有效。
- 用标准库实现一个玩具版 HTN 规划器和一个玩具版进化搜索。

## 问题背景

ReWOO（第 02 课）、Plan-and-Execute 和 ReAct 已经覆盖了大部分智能体规划场景。但有两类情况它们处理不好：

1. **需要可证明正确性的计划。** 排程、航线规划、合规流程——计划必须在构造上就是可靠的。一个表达流畅、却偶尔幻觉出某个步骤的 LLM 计划是不可接受的。
2. **拥有机器可校验适应度函数的优化问题。** 矩阵乘法、调度启发式、编译器优化——目标不是「一个正确的计划」，而是「最优的计划」。

HTN 规划和 AlphaEvolve 分别解决这两个不同的问题。两者都把 LLM 当作放大器，而不是替代品。

## 核心概念

### 分层任务网络

一个 HTN 包含：

- **任务（Tasks）**——分为复合任务（待分解）和原子任务（可直接执行）。
- **方法（Methods）**——把复合任务分解为子任务的途径，带有前置条件。
- **算子（Operators）**——带前置条件和效果的原子动作。
- **状态（State）**——一组事实。

规划过程：给定目标任务和初始状态，找到一个分解方案，使其落到一串原子算子上，且这些算子的前置条件按顺序依次满足。

HTN 比 LLM 古老得多，至今仍是可证明正确计划的标杆方案。

### ChatHTN（Gopalakrishnan et al., 2025）

ChatHTN（arXiv:2505.11814）将符号 HTN 与 LLM 查询交织在一起：

1. 先尝试用已有方法分解当前复合任务。
2. 如果没有方法适用，就问 LLM：「在状态 `s` 下你会怎么分解 `task`？」
3. 将 LLM 的回答转换为候选子任务。
4. 对照算子模式（schema）做校验，拒绝无效的分解。
5. 递归执行。

论文的核心论断：产出的每个计划都是可证明可靠的，因为 LLM 的建议只能以候选分解的形式进入系统，永远无法直接修改计划。符号层负责正确性，LLM 负责扩充方法库。

在线方法学习（OpenReview `gwYEDY9j2x`，2025 年后续工作）增加了一个学习器，通过回归泛化 LLM 产生的分解——可把 LLM 查询频率降低多达 75%。

### AlphaEvolve（Novikov et al., 2025）

AlphaEvolve（arXiv:2506.13131，DeepMind，2025 年 6 月）是完全不同的路数：由 Gemini 2.0 Flash/Pro 集成模型编排的进化代码搜索。

循环过程：

1. 从一个种子程序 + 一个可编程评估器（返回适应度分数）开始。
2. LLM 集成提出变异。
3. 把变异送进评估器运行。
4. 保留最优者，继续变异。

已发表的成果：

- 56 年来首次在 4x4 复数矩阵乘法上超越 Strassen 算法（48 次标量乘法）。
- 通过一个 Borg 调度启发式回收了 Google 0.7% 的算力。
- 在一个前沿工作负载上让 FlashAttention 提速 32%。

硬性约束：适应度函数必须可由机器校验。对自由文本答案做进化搜索是不会收敛的。

### 何时用哪个

| 问题类别 | 用什么 | 原因 |
|---------------|-----|-----|
| 带硬约束的排程 | HTN + ChatHTN | 可证明的可靠性 |
| 编译器优化 | AlphaEvolve | 机器可校验的适应度 |
| 多步任务执行 | ReAct / ReWOO | LLM 在循环中，无形式化保证 |
| 带测试的代码改进 | AlphaEvolve | 测试就是评估器 |
| 受策略约束的自动化 | HTN | 前置条件编码了策略 |

### 这种模式容易出错的地方

- **没有算子的 HTN。** 缺少前置条件/效果模式，可靠性的论断就会崩塌。ChatHTN 的「LLM 建议分解」必须依靠模式来拒绝无效的操作。
- **没有真实评估器的 AlphaEvolve。** 「问 LLM 代码是不是更好了」不是适应度函数。评估器必须确定且快速。
- **过度工程。** 大多数智能体任务两者都不需要。优先考虑 ReAct 或 ReWOO。

## 从零实现

`code/main.py` 实现了两个玩具示例：

- 一个标准库实现的 HTN 规划器，包含算子、方法、前置条件、效果，以及一个在没有方法匹配复合任务时介入的 `LLMFallback`。这里的「LLM」是一个脚本化的分解器，所以规划器可以离线运行。
- 一个标准库实现的算术程序进化搜索：生长表达式，使其输出在测试集上最小化 `|f(x) - target|`。评估器是确定性的。

运行方式：

```
python3 code/main.py
```

运行轨迹会展示 HTN 规划器分解复合任务的过程（中途触发一次 LLM 兜底），以及进化循环收敛到目标表达式的过程。

## 生产实践

- **HTN 规划器**——`pyhop`、`SHOP3`，或者为特定领域的策略约束自己实现一个。
- **ChatHTN**——研究代码；其模式（符号规划 + LLM 兜底）可以干净地移植到任何 HTN 规划器。
- **AlphaEvolve**——DeepMind 论文；其模式（集成模型 + 评估器）是可复现的。OpenEvolve 等开源复刻正在涌现。
- **智能体框架**——目前没有任何框架原生提供 HTN 或 AlphaEvolve。把它做成子智能体或后台 worker。

## 练习

1. 给 HTN 规划器加上回溯：当某个算子的后置条件在运行时不满足，回滚并尝试下一个方法。
2. 为 ChatHTN 添加 LLM 方法缓存：当 LLM 在状态模式 `P` 下分解了任务 `T`，存储结果。下次调用时先检查方法库。
3. 把进化搜索的评估器换成一个真实测试套件。进化出一个通过 20 个测试用例的排序函数，并报告收敛所需的代数。
4. 阅读 AlphaEvolve 的评估器设计笔记。为你关心的领域设计一个评估器（SQL 查询优化、测试套件最小化、部署 YAML）。
5. 组合使用：先用 HTN 把复合任务分解为子任务，再对每个子任务的原子算子做进化搜索。它在哪些地方有奇效，在哪些地方是过度工程？

## 关键术语

| 术语 | 通俗说法 | 实际含义 |
|------|----------------|------------------------|
| HTN | 「分层规划器」 | 带算子、前置条件、效果的任务分解 |
| 方法（Method） | 「分解规则」 | 把复合任务拆成子任务的途径 |
| 算子（Operator） | 「原子动作」 | 带前置条件和效果的具体步骤 |
| ChatHTN | 「LLM + HTN」 | 没有方法匹配时，符号规划器向 LLM 求助 |
| AlphaEvolve | 「进化代码搜索」 | LLM 集成变异代码，确定性评估器做选择 |
| 适应度函数（Fitness function） | 「评估器」 | 对输出的确定性、机器可校验的打分 |
| 在线方法学习 | 「缓存 LLM 分解」 | 存储并泛化 LLM 计划以降低查询成本 |

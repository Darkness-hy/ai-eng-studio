# 评测驱动的智能体开发

> Anthropic 的指导原则："从简单的提示词开始，用全面的评测来优化它们，只在必要时才引入多步骤的智能体系统。"评测不是最后一步，而是驱动 Phase 14 中所有其他选择的外层循环。

**Type:** Learn + Build
**Languages:** Python (stdlib)
**Prerequisites:** All of Phase 14.
**Time:** ~60 minutes

## 学习目标

- 说出三个评测层次——静态基准、自定义离线评测、线上生产评测——以及它们各自的用途。
- 解释评估器-优化器（evaluator-optimizer）紧密循环。
- 描述 2026 年的最佳实践：评测与代码放在一起、在 CI 中运行、把关 PR。
- 把 Phase 14 的每节课与它产生的评测用例联系起来。

## 问题背景

智能体能通过演示，却会以演示无法预测的方式在生产环境中失败。基准测试回答的是"这个模型是否具备广泛能力？"，而不是"这个智能体是否在为我的产品交付正确的补丁？"答案是：在三个层次上持续运行评测，并把每条护栏和每条习得规则都映射到一个评测用例。

## 核心概念

### 三个评测层次

1. **静态基准（static benchmarks）**——代码用 SWE-bench Verified（第 19 课），浏览/桌面用 WebArena/OSWorld（第 20 课），通用任务用 GAIA（第 19 课），工具使用用 BFCL V4（第 06 课）。用于跨模型对比和回归把关。数据污染是真实存在的：SWE-bench+ 发现 32.67% 的解答泄漏。务必报告 Verified 或经 + 审计的分数。

2. **自定义离线评测（custom offline evals）**——贴合你产品的形态：
   - LLM 充当评审（LLM-as-judge）（Langfuse、Phoenix、Opik——第 24 课）。
   - 基于执行的评测（运行补丁、检查测试）。
   - 基于轨迹的评测（将动作序列与黄金标准对比；OSWorld-Human 显示顶尖智能体的步数是黄金轨迹的 1.4-2.7 倍）。

3. **线上评测（online evals）**——生产环境：
   - 会话回放（Langfuse）。
   - 护栏触发的告警（第 16、21 课）。
   - 逐步的成本/延迟跟踪（第 23 课 OTel span）。

### 评估器-优化器（Anthropic）

紧密循环：

1. 提议者（proposer）生成输出。
2. 评估器（evaluator）做出评判。
3. 不断改进，直到评估器通过。

这就是 Self-Refine（第 05 课）的泛化形式。任何你关心的智能体流程都可以包裹在评估器-优化器中以提高可靠性。

### 2026 年最佳实践

- 评测与代码放在一起。
- 在 CI 中对每个 PR 运行。
- 用评测分数把关合并（例如"相对 main 的回退不超过 5%"）。
- 每条护栏都映射到一个评测用例。
- 每条习得规则（Reflexion、pro-workflow 的 learn-rule）都映射到一个失败用例。

### 评测驱动开发会在哪里失败

- **没有基线。** 没有"最近已知良好"参照的评测无从解读。要存储基线。
- **LLM 评审缺乏依据。** 评审同样会产生幻觉。用 CRITIC 模式（第 05 课）——让评审基于外部工具进行验证。
- **对评测过拟合。** 针对评测做优化会偏离生产环境中的实际有用性。要轮换用例。
- **不稳定的评测。** 非确定性的用例会引发误报。固定随机种子、对状态做快照。

## 从零实现

`code/main.py` 是一个纯标准库实现的评测框架：

- 带分类（benchmark、custom、online）的用例注册表。
- 一个被测的脚本化智能体。
- 评估器-优化器循环：提议、评判、改进，直到通过或达到最大轮数。
- CI 门禁：汇总通过率 + 与基线的回归对比。

运行：

```
python3 code/main.py
```

输出：每个用例的通过/失败、回归标记、CI 门禁裁决。

## 生产实践

- 把评测用例写在与智能体代码相同的仓库里。
- 通过 CI 在每个 PR 上运行。
- 出现回归时让构建失败。
- 持续跟踪通过率随时间的变化。
- 把每一次生产故障都对应到一个新的评测用例。

## 交付产物

`outputs/skill-eval-suite.md` 为一个智能体产品构建带 CI 门禁和回归跟踪的三层评测套件。

## 练习

1. 拿一个你的生产故障，写一个能复现它的评测用例。你的智能体现在能通过吗？
2. 为你的领域构建一个包含三个维度（事实性、语气、范围）的 LLM 评审打分标准，并对 50 个会话打分。
3. 把评测套件接入 CI。回归 >=5% 时让构建失败。
4. 增加一个轨迹效率指标：相比黄金轨迹，智能体用了多少步？
5. 把 Phase 14 的每节课映射到你套件中的一个评测用例。有缺失吗？那就是要补的缺口。

## 关键术语

| 术语 | 人们常说 | 实际含义 |
|------|----------------|------------------------|
| 静态基准 | "现成的评测" | SWE-bench、GAIA、AgentBench、WebArena、OSWorld |
| 自定义离线评测 | "领域评测" | 针对你产品形态的 LLM 评审 / 执行 / 轨迹评测 |
| 线上评测 | "生产评测" | 会话回放、护栏告警、成本/延迟跟踪 |
| 评估器-优化器 | "提议-评判-改进" | 迭代直到评审通过 |
| CI 门禁 | "合并拦截器" | 评测回归时让构建失败 |
| 基线 | "最近已知良好" | 用于检测回归的参考分数 |
| 轨迹效率 | "相对黄金轨迹的步数" | 智能体步数除以人类专家的最少步数 |

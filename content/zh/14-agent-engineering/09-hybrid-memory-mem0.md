# 混合记忆：向量 + 图 + KV（Mem0）

> Mem0（Chhikara et al., 2025）把记忆视为三个并行的存储——向量存储负责语义相似度，KV 存储负责快速事实查询，图存储负责实体关系推理。检索时由一个评分层将三者融合。这是 2026 年外部记忆的生产标准方案。

**Type:** Build
**Languages:** Python (stdlib)
**Prerequisites:** Phase 14 · 07 (MemGPT), Phase 14 · 08 (Letta Blocks)
**Time:** ~75 minutes

## 学习目标

- 解释为什么单一存储（只用向量、只用图、或只用 KV）不足以支撑智能体记忆。
- 说出 Mem0 的三个并行存储，以及每个存储各自针对什么场景做优化。
- 描述 Mem0 的融合评分——相关性、重要性、新近度——并解释它为什么是加权求和而不是层级结构。
- 用标准库实现一个玩具版三存储记忆：`add()` 同时写入三个存储，`search()` 融合三者的结果。

## 问题背景

无论选哪种单一存储，三类查询中总有一类它做不好：

- **语义相似度**——"上周我们讨论过哪些关于智能体漂移的内容？"向量存储胜出；KV 和图都会漏掉。
- **事实查询**——"用户的电话号码是什么？"KV 胜出；用向量太浪费，用图则是杀鸡用牛刀。
- **关系推理**——"哪些客户共用同一个账单实体？"图存储胜出；向量和 KV 根本回答不了。

生产环境的智能体在一个会话里三类查询都会出现。单一存储的记忆系统总会在其中两类上失败。Mem0 的贡献在于把三个存储统一接到同一个 `add`/`search` 接口背后，并用一个评分函数把它们融合起来。

## 核心概念

### 三个并行存储

Mem0（arXiv:2504.19413，2025 年 4 月）在 `add(text, user_id, metadata)` 时：

1. 从文本中抽取候选事实（由 LLM 驱动的步骤）。
2. 把每条事实写入向量存储（嵌入），用于语义搜索。
3. 把每条事实写入 KV 存储，以 (user_id, fact_type, entity) 为键，实现 O(1) 查询。
4. 把每条事实以带类型的边写入图存储（Mem0g），用于关系查询。

在 `search(query, user_id)` 时：

1. 向量存储按嵌入余弦相似度返回 top-k。
2. KV 存储按从查询推导出的 (user_id, type, entity) 键返回精确命中。
3. 图存储返回从查询实体可达的子图。
4. 评分层将三者的结果融合。

### 融合评分

```
score = w_relevance * relevance(q, record)
      + w_importance * importance(record)
      + w_recency * recency(record)
```

- **相关性（Relevance）**——向量存储用余弦相似度，KV 用精确匹配，图用路径权重。
- **重要性（Importance）**——在写入时打标或通过学习获得（有些事实更重要：姓名、ID、政策条款）。
- **新近度（Recency）**——按最近一次写入或读取以来的时间做指数衰减。

权重按产品调优。聊天智能体调高 `w_recency`；合规智能体调高 `w_importance`；检索智能体调高 `w_relevance`。

### Mem0g 与时序推理

Mem0g 增加了一个冲突检测器。当新事实与已有的边矛盾时，已有的边会被标记为失效，但不会删除。时序查询（"用户三月份住在哪个城市？"）会遍历在指定时间点有效的子图。

这正是 Letta 失效标记模式所泛化出的合规级行为。

### 基准测试数据

Mem0 论文（2025）报告的结果：

- **LoCoMo**（长篇对话记忆）：91.6
- **LongMemEval**（长程情景记忆）：93.4
- **BEAM 1M**（百万 token 记忆基准）：64.1

对比基线（128k 全上下文 LLM、扁平向量存储、扁平 KV）全部落后 10 分以上。基准分数本身不足以决定选型——工程形态才是关键——但这些数字说明融合设计带来的提升绝非误差范围内的波动。

### 作用域分类

Mem0 按作用域划分记忆：

- **用户记忆**——跨会话持久化，以 `user_id` 为键。
- **会话记忆**——在单个线程内持久化。
- **智能体记忆**——每个智能体实例自己的状态。

每次写入都要选定一个作用域。检索可以跨作用域查询，并对每个作用域使用不同权重。不假思索地混用作用域，正是"助手把 Bob 的项目告诉了 Alice"这类事故的根源。

### 这个模式会在哪里出问题

- **嵌入漂移。**前一百次查询看起来正确的向量检索结果，会随着语料库增长而退化。应定期对使用频率最高的 N 条记录重新做嵌入。
- **KV schema 膨胀。**`(user_id, type, entity)` 看似简单，直到每个团队都开始添加自己的 `type`。每季度审计一次类型集合。
- **图爆炸。**一个噪声大的抽取器每条消息能加 50 条边。要限制每次 `add` 调用的图写入量，并丢弃低置信度的边。

## 从零实现

`code/main.py` 用标准库实现了三存储模式：

- `VectorStore`——用朴素的 token 重叠相似度代替真实嵌入。
- `KVStore`——以 `(user_id, fact_type, entity)` 为键的字典。
- `GraphStore`——带类型的边 (subject, relation, object, valid)。
- `Mem0`——顶层门面类，提供 `add()`、`search()`、融合评分和按作用域感知的检索。
- 在一段多用户、多会话对话上的完整运行轨迹。

运行：

```
python3 code/main.py
```

输出会展示三条独立的召回路径，以及融合后的 top-k。修改 `main()` 顶部的评分权重，观察排序如何变化。

## 生产实践

- **Mem0（Apache 2.0）**——可直接用于生产。自托管可选 Postgres + Qdrant + Neo4j，也可使用托管云服务。
- **Letta**——core/recall/archival 三层结构；向量和图后端需自行接入。
- **Zep**——商业替代方案，带时序知识图谱和事实抽取。
- **自研方案**——当你需要完全掌控抽取器（合规场景）或融合权重（语音智能体场景，新近度占主导）时。

## 练习

1. 用真实的嵌入模型（sentence-transformers、Ollama、OpenAI embeddings）替换玩具版向量相似度。在一段合成的长对话上测量 recall@10。经过 1000 次写入后排序会漂移吗？
2. 添加一个时序查询：`search(query, as_of=timestamp)`。只返回在该时间点或之前有效的记录。哪个存储需要的改动最大？
3. 实现一个冲突检测器：当新事实与某条图边矛盾时，将旧边标记为失效并把两者都记入日志。用"用户住在 Berlin"->"用户住在 Lisbon"测试。
4. 给融合评分器增加一个 `user_feedback` 维度（对检索结果点赞）。如何防止被刷分（智能体只返回它已经"喜欢过"的记录）？
5. 阅读 Mem0 文档（`docs.mem0.ai`）。把玩具版改写成 `mem0` 客户端调用。在同样的 20 条测试查询上比较检索质量。

## 关键术语

| 术语 | 人们怎么说 | 实际含义 |
|------|----------------|------------------------|
| 混合记忆 | "向量加图加 KV" | 三个存储并行写入，检索时融合 |
| 事实抽取 | "记忆摄入" | 由 LLM 把文本拆解为 (entity, relation, fact) 三元组的步骤 |
| 融合评分 | "相关性排序" | 相关性、重要性、新近度的加权求和 |
| 作用域 | "记忆命名空间" | user / session / agent——决定谁能看到什么 |
| Mem0g | "记忆图" | 带时间有效性的类型化边，用于关系查询 |
| 时序失效 | "软删除" | 把被矛盾覆盖的边标记为失效；永不删除 |
| 嵌入漂移 | "检索腐化" | 向量质量随语料库增长而退化；需定期重新嵌入 |
